# ERP QA Kit v0.2 — Functional Specification

**Status:** Planning (PLANNING ONLY — no code in this doc).
**Audience:** Codex (implementation), PM (review).
**Sibling docs (do not duplicate here):** `v0.2_PROJECT_POLICY_SPEC.md`, `v0.2_PROJECT_MEMORY_SPEC.md`, `v0.2_EXTRACTOR_STRATEGY.md`, `v0.2_ARCHITECTURE.md`, `v0.2_CONSTRAINTS.md`, `v0.2_ACCEPTANCE_CRITERIA.md`.

---

## 0. Core principle (binding on all sections)

LLMs **propose**; deterministic extractors/checks **verify**; humans **approve**. v0.2 never modifies target ERP code, never executes DB queries, and writes **only** under `qa-context/`. The frontend is the **implementation under test, not the source of truth**. Source of truth for visible text / labels / columns / search filters / form fields = **screen spec + backend evidence**. FRONTEND OVERRIDE IS FORBIDDEN where policy says so.

---

## 1. Scope & relationship to v0.1 (additive only)

v0.1 shipped: Python ≥ 3.11; sole runtime dependency PyYAML; `argparse`-based `erpqa` CLI where each subcommand takes a positional `project_path`; the write-containment guard `assert_within_qa_context`; and provenance constants.

v0.2 is **strictly additive**. It MUST NOT break v0.1:

| Constraint | v0.2 obligation |
|---|---|
| Python / deps | Stay ≥ 3.11; PyYAML remains the only **required** dep. Extractor libs (openpyxl) and OPTIONAL libs (pandas, markitdown) are lazy/optional — see `v0.2_EXTRACTOR_STRATEGY.md`. |
| CLI shape | Reuse `erpqa <subcommand> project_path` argparse pattern. New subcommands add a `--module <NAME>` option (except `policy-init`, which is project-level). |
| Write containment | Every write goes through `assert_within_qa_context`. No writes outside `qa-context/`. |
| Provenance | Reuse v0.1 provenance constants in every contract `source`/`extraction_method` field. |
| Exit codes | Reuse v0.1 contract: `0` ok; non-zero for validation / usage / safety errors (see §2.0). |
| Existing v0.1 files | Not modified by this planning doc; v0.2 code extends, does not rewrite. |

Anti-overengineering (binding): when a `--module` is given, **no full-repo analysis**; **no required cloning of all repos**; module-scoped extracted folders are preferred. Keep it simple. Required extractors must be **real + wired + tested** (see `v0.2_EXTRACTOR_STRATEGY.md`).

## 2. New CLI commands

### 2.0 Shared exit-code contract (reused by all commands)

| Code | Meaning |
|---|---|
| `0` | Success (work done, or idempotent no-op). |
| `1` | Validation error (bad/missing contract, schema violation, malformed YAML). |
| `2` | Usage error (missing `--module`, unknown command, missing `project_path`). |
| `3` | Safety violation (attempted write outside `qa-context/`, attempted ERP-code mod, attempted DB execution). |

**Universal preconditions (all commands):** Each command reads **project policy + project memory FIRST**, and (for module commands) **module policy + module memory**, BEFORE doing any work. See §9 for the per-command read table. If required policy is missing, behavior follows §9.

All module commands are **idempotent / no-clobber**: re-running regenerates derived files in place but never silently destroys human-confirmed edits — confirmation flags and human-edited memory are preserved (see §8). Generated reports/handoffs are overwritten; contracts are merged additively where a `needs_human_confirmation: false` decision exists.

---

### 2.1 `policy-init` (project-level)

- **Synopsis:** `erpqa policy-init <project_path>` (NO `--module`).
- **Reads first:** existing `qa-context/project_policy.yaml`, `project_memory.md`, `project_assumptions.yaml` if present.
- **Writes:** `qa-context/project_policy.yaml`, `qa-context/project_memory.md`, `qa-context/project_assumptions.yaml` (scaffolds). Schema/content defined in `v0.2_PROJECT_POLICY_SPEC.md` and `v0.2_PROJECT_MEMORY_SPEC.md`.
- **Behavior:** Scaffolds project-level policy including `validation_order`, `defer_until_frontend_verified`, `frontend_override_forbidden`, source-of-truth precedence, and pilot constraints (7 modules, 3 repos incl. ORD).
- **Idempotency / no-clobber:** If files exist, does NOT overwrite human content; reports they exist and exits `0`. Use a future `--force` only to re-scaffold (not required for v0.2).
- **Exit codes:** `0` created or already present; `2` missing `project_path`; `3` path outside containment.
- **Failure modes:** unwritable path → `1`; outside `qa-context/` → `3`.

### 2.2 `module-init`

- **Synopsis:** `erpqa module-init <project_path> --module <NAME>`.
- **Reads first:** project policy + project memory; then existing module files if present.
- **Writes:** `qa-context/modules/<NAME>/{module_manifest.yaml, module_policy.yaml, module_memory.md}`.
- **Behavior:** Creates the module folder + scaffolds manifest (declares which extracted source folders/files belong to this module), module policy (overrides/inherits project policy), and module memory.
- **Idempotency / no-clobber:** Existing human-edited files preserved; missing ones scaffolded. Exit `0`.
- **Exit codes:** `0`; `2` missing `--module`/`project_path`; `1` project policy missing & required (per §9); `3` containment.
- **Failure modes:** module name with path separators → `2`; project policy absent → see §9.

### 2.3 `extract-spec`

- **Synopsis:** `erpqa extract-spec <project_path> --module <NAME>`.
- **Reads first:** project + module policy & memory; `module_manifest.yaml` to locate spec source files (xlsx/csv/markdown/json/yaml).
- **Writes:** `qa-context/modules/<NAME>/screen_contract.yaml`.
- **Behavior:** Runs the appropriate extractor(s) (xlsx_text_adapter, xlsx_image_manifest_adapter, csv_adapter, markdown_adapter, json_yaml_adapter; OPTIONAL markitdown) over declared spec files, normalizes into `screen_contract.yaml` (§4) with per-item `source`/`confidence`/`needs_human_confirmation`. This is the **source-of-truth** screen contract.
- **Idempotency / no-clobber:** Regenerates items, but preserves any item where `needs_human_confirmation: false` was set by a human (merge by `key`). Exit `0`.
- **Exit codes:** `0`; `1` no spec source declared or unparseable; `2` usage; `3` containment.
- **Failure modes:** image-derived/OCR-style content → confidence capped low/medium, `needs_human_confirmation: true` always (§8).

### 2.4 `extract-frontend`

- **Synopsis:** `erpqa extract-frontend <project_path> --module <NAME>`.
- **Reads first:** project + module policy & memory; manifest to locate module-scoped frontend extracted folder.
- **Writes:** `qa-context/modules/<NAME>/frontend_contract.yaml`.
- **Behavior:** `frontend_static_adapter` performs **static** analysis (no execution): detects routes/components, visible text, search filters, grid columns, form fields, buttons/actions, hidden/internal fields, optional API calls. Frontend is implementation-under-test — never written as truth.
- **Idempotency / no-clobber:** Regenerates; preserves human confirmations by `key`. Exit `0`.
- **Exit codes:** `0`; `1` no frontend source declared; `2` usage; `3` containment.
- **Failure modes:** inferred mappings → medium/low confidence + `needs_human_confirmation: true`.

### 2.5 `extract-backend`

- **Synopsis:** `erpqa extract-backend <project_path> --module <NAME>`.
- **Reads first:** project + module policy & memory; manifest to locate module-scoped backend extracted folder.
- **Writes:** `qa-context/modules/<NAME>/backend_contract.yaml`.
- **Behavior:** `backend_static_adapter` (static, no execution): endpoints, DTO request/response fields, service/repository files, optional procedure calls, field-mapping evidence. Backend is **corroborating evidence** for source of truth (per precedence), but not fully trusted.
- **Idempotency / no-clobber:** Regenerates; preserves confirmations. Exit `0`.
- **Exit codes:** `0`; `1` no backend source; `2` usage; `3` containment.
- **Failure modes:** inferred field mappings → medium/low + `needs_human_confirmation: true`.

### 2.6 `extract-procedure`

- **Synopsis:** `erpqa extract-procedure <project_path> --module <NAME>`.
- **Reads first:** project + module policy & memory; manifest to locate stored-procedure SQL files.
- **Writes:** `qa-context/modules/<NAME>/procedure_contract.yaml`.
- **Behavior:** `procedure_sql_adapter` parses SQL **statically, NO DB execution**: procedure name, file path, parameters, optional result columns, optional tables touched. Client stored procedures may reflect *intended* logic but are **unverified**.
- **Deferral:** Procedure / business-logic validation is **DEFERRED** until frontend mismatches are resolved (`defer_until_frontend_verified`). This command may still *extract* (cheap, safe), but downstream procedure validation is gated in `module-audit`.
- **Idempotency / no-clobber:** Regenerates; preserves confirmations. Exit `0`.
- **Exit codes:** `0`; `1` unparseable SQL / no procedure source; `2` usage; `3` containment. **Never** connects to a DB — any DB-connection attempt is a `3` safety violation by design.
- **Failure modes:** ambiguous SQL → low confidence + `needs_human_confirmation: true`.

### 2.7 `compare-contract`

- **Synopsis:** `erpqa compare-contract <project_path> --module <NAME>`.
- **Reads first:** project + module policy & memory; then `screen_contract.yaml`, `frontend_contract.yaml`, `backend_contract.yaml` (and `procedure_contract.yaml` only if frontend already verified per policy).
- **Writes:** intermediate comparison findings consumed by `module-report` (materialized into the reports in §6; may persist a machine-readable findings file alongside reports — implementation detail, kept simple).
- **Behavior:** Runs the comparison logic of §5. **Source-of-truth precedence = `screen_spec` then `backend`** (per policy). Frontend never overrides when `frontend_override_forbidden: true`. Detects all mismatch types in §5.
- **Idempotency / no-clobber:** Pure derivation from contracts; safe to re-run. Exit `0`.
- **Exit codes:** `0` (even if mismatches found — mismatches are data, not errors); `1` missing required contract (e.g., no `screen_contract.yaml`); `2` usage; `3` containment.
- **Failure modes:** if either side of a comparison is low confidence, the finding's `needs_human_confirmation` propagates `true` (§5).

### 2.8 `module-report`

- **Synopsis:** `erpqa module-report <project_path> --module <NAME>`.
- **Reads first:** project + module policy & memory; comparison findings + all contracts.
- **Writes:** `qa-context/modules/<NAME>/reports/{screen_frontend_mismatch_report.md, column_mismatch_report.md, frontend_contract_report.md, procedure_mapping_report.md}`.
- **Behavior:** Renders PM-readable reports per §6. The mismatch report includes every per-finding field enumerated in §6 (incl. `frontend_override_forbidden`, `backend_evidence_agrees_or_conflicts`, `project_memory_read`, `module_memory_read`, `handoff_ready_ai_fix_instruction`). `procedure_mapping_report.md` is generated but flagged DEFERRED until frontend verified.
- **Idempotency / no-clobber:** Reports are regenerated (overwritten) each run. Exit `0`.
- **Exit codes:** `0`; `1` missing comparison/contracts; `2` usage; `3` containment.
- **Failure modes:** no findings → report states "no mismatches detected at current confidence" and still records memory-read booleans.

### 2.9 `module-handoff`

- **Synopsis:** `erpqa module-handoff <project_path> --module <NAME>`.
- **Reads first:** project + module policy & memory; mismatch report + contracts.
- **Writes:** `qa-context/modules/<NAME>/handoff/{frontend_fix_handoff.md, codex_frontend_fix_prompt.md}`.
- **Behavior:** Produces the human handoff and the Codex fix prompt per §7. Instructs fixing the **TARGET FRONTEND repo**, NOT ERP QA Kit; explicitly states project_memory.md + module_memory.md were read; reminds that frontend override is forbidden; **no auto-modification**.
- **Idempotency / no-clobber:** Regenerated each run. Exit `0`.
- **Exit codes:** `0`; `1` no mismatch report present; `2` usage; `3` containment.
- **Failure modes:** no actionable frontend findings → handoff states nothing to fix yet.

### 2.10 `module-audit` (orchestrator)

- **Synopsis:** `erpqa module-audit <project_path> --module <NAME>`.
- **Reads first:** project + module policy & memory (incl. `validation_order` and `defer_until_frontend_verified`).
- **Writes:** every output of the orchestrated steps (contracts, reports, handoffs).
- **Behavior:** Runs granular commands in the order declared by policy `validation_order`. Default orchestrated order:

  1. `extract-spec`
  2. `extract-frontend`
  3. `extract-backend`
  4. `extract-procedure` *(extraction only; safe)*
  5. `compare-contract`
  6. `module-report`
  7. `module-handoff`

  **`defer_until_frontend_verified` honoring:** while frontend mismatches against source of truth remain **unresolved/unconfirmed**, `module-audit` SKIPS procedure/business-logic *validation and mapping conclusions* (procedure extraction may run, but `procedure_mapping_report.md` is emitted as DEFERRED and not treated as authoritative). FIRST priority is frontend screen correctness.
- **Idempotency / no-clobber:** Equivalent to running steps individually; preserves human confirmations and memory edits. Exit `0`.
- **Exit codes:** `0` if all steps succeed (mismatches are not failures); first non-zero step code propagates (`1`/`2`/`3`).
- **Failure modes:** a missing required contract halts at that step with `1`; usage/containment as above.

## 4. Contract schemas

All contracts: required fields plus `source`, `confidence` (`high|medium|low`), `needs_human_confirmation` (bool). `source` / `extraction_method` reuse v0.1 provenance constants. Placeholder names are generic.

### 4.1 `screen_contract.yaml` (SOURCE OF TRUTH)

**Required:** `screen_id`, `screen_name`, `module`, `source_files`, `extraction_method`, `confidence`, `needs_human_confirmation`, `sections{visible_text, search_filters, grid_columns, form_fields, buttons_actions, hidden_fields}`.
**Each item under a section:** `key`, `label`, `visible`, `required`, `source`, `confidence`, `needs_human_confirmation`; optional `readonly_disabled`, `order`, `data_type`, `code_combo_source`.

```yaml
screen_id: SCR_0001
screen_name: Example List Screen
module: MODULE_A
source_files:
  - extracted/MODULE_A/spec/screen_spec.xlsx
extraction_method: xlsx_text_adapter
confidence: medium
needs_human_confirmation: true
sections:
  visible_text:
    - key: title_label
      label: "Order Inquiry"
      visible: true
      required: false
      order: 1
      source: screen_spec
      confidence: medium
      needs_human_confirmation: true
  search_filters:
    - key: order_date
      label: "Order Date"
      visible: true
      required: false
      data_type: date
      order: 1
      source: screen_spec
      confidence: medium
      needs_human_confirmation: true
  grid_columns:
    - key: order_no
      label: "Order No."
      visible: true
      required: true
      order: 1
      data_type: string
      source: screen_spec
      confidence: medium
      needs_human_confirmation: true
  form_fields:
    - key: customer_code
      label: "Customer"
      visible: true
      required: true
      readonly_disabled: false
      data_type: string
      code_combo_source: CUSTOMER_MASTER
      source: screen_spec
      confidence: medium
      needs_human_confirmation: true
  buttons_actions:
    - key: btn_search
      label: "Search"
      visible: true
      required: false
      source: screen_spec
      confidence: medium
      needs_human_confirmation: true
  hidden_fields:
    - key: internal_seq
      label: "Internal Sequence"
      visible: false
      required: false
      source: screen_spec
      confidence: medium
      needs_human_confirmation: true
```

```yaml
    visible: true
```

### 4.3 `backend_contract.yaml` (corroborating evidence)

**Required:** `endpoints`, `dto_request_response_fields`, `service_repository_files`, `field_mapping_evidence`, `source`, `confidence`, `needs_human_confirmation`. **Optional:** `procedure_calls`.

```yaml
  - dbo.orders
  - dbo.customers

needs_human_confirmation: true   # client procedure: intended-but-unverified
```

## 5. Comparison logic spec

**Inputs:** `screen_contract.yaml` (truth, precedence 1) + `backend_contract.yaml` (corroboration, precedence 2) vs `frontend_contract.yaml` (under test).
**Precedence:** `screen_spec` then `backend`. **Frontend never overrides** when `frontend_override_forbidden: true`.
**Confidence propagation (applies to every row):** if **either** compared side is `low` confidence (or carries `needs_human_confirmation: true`), the resulting finding sets `needs_human_confirmation: true` and confidence is capped at the **lower** of the two sides. A finding may never be emitted as authoritative if it rests on low-confidence-only evidence.

| Mismatch type | Definition | How detected from contracts | Default severity | needs_human_confirmation propagation |
|---|---|---|---|---|
| missing FE columns | A truth grid column absent in FE | `key` in `screen.grid_columns` but not in `frontend.grid_columns` | high | true if either side low conf |
| unexpected FE columns | FE shows a column not in truth | `key` in `frontend.grid_columns` not in `screen.grid_columns` (and not justified by backend) | high (data exposure risk) | true if FE low conf or no backend evidence |
| label mismatch | Visible label differs from truth | same `key`, `label` differs (screen vs FE) | medium | true if either side low conf |
| order mismatch | Display order differs from truth | same `key`, `order` differs | low | true if either side low conf |
| visibility mismatch | `visible` differs from truth | same `key`, `visible` differs | medium | true if either side low conf |
| required/optional mismatch | `required` differs from truth | same `key`, `required` differs | medium | true if either side low conf |
| readonly/disabled mismatch | `readonly_disabled` differs from truth | same `key`, `readonly_disabled` differs | medium | true if either side low conf |
| field-key mismatch | FE key not matching any truth key | FE `key` has no truth counterpart (and vice versa) | high | true if either side low conf |
| search-filter mismatch | Filter set/labels differ from truth | diff of `search_filters` by `key`/`label`/`visible` | medium | true if either side low conf |
| grid-column mismatch | Aggregate column-set differences | union diff of `grid_columns` (covers missing/unexpected/order) | high | true if either side low conf |
| form-field mismatch | Form field set/attrs differ from truth | diff of `form_fields` by `key`/`label`/`required`/`readonly_disabled` | medium | true if either side low conf |
| buttons/actions mismatch | Buttons/actions differ from truth | diff of `buttons_actions` by `key`/`label`/`visible` | medium | true if either side low conf |
| internal/audit/system fields exposed | FE exposes hidden/internal fields not allowed | truth `hidden_fields` (`visible:false`) appears in `frontend.*` with `visible:true`, OR FE `hidden_internal_fields` visible | high | true if FE low conf |
| FE overriding screen-spec labels/definitions | FE asserts a label/definition that contradicts truth | same `key`, FE value differs AND `frontend_override_forbidden:true` | high | true if either side low conf; finding flagged `frontend_override_forbidden:true` |

Backend role: when backend evidence agrees with truth, it **strengthens** a finding (lowers `needs_human_confirmation` only if both truth and backend are high); when backend **conflicts** with truth, the finding records `backend_evidence_agrees_or_conflicts: conflicts` and remains human-confirm-required. Backend never outranks `screen_spec`.

## 6. Report spec

`module-report` writes four reports. All are PM-readable Markdown.

| Report | Contains |
|---|---|
| `screen_frontend_mismatch_report.md` | The primary PM mismatch report — one entry per finding (fields below). |
| `column_mismatch_report.md` | Grid/column-focused subset: missing/unexpected/order/label column mismatches, table form. |
| `frontend_contract_report.md` | What the FE actually exposes vs source of truth (inventory + deltas), incl. exposed internal fields. |
| `procedure_mapping_report.md` | Procedure → result-columns → truth mapping; **flagged DEFERRED** until frontend verified; not authoritative. |

**Per-finding fields (REQUIRED in `screen_frontend_mismatch_report.md`):**

| Field | Meaning |
|---|---|
| `screen_id` | Screen identifier. |
| `module` | Module name. |
| `mismatch_type` | One of §5 types. |
| `expected_from_source_of_truth` | Value per screen_spec (then backend). |
| `actual_frontend` | Value found in FE. |
| `source_files` | Files backing both sides. |
| `confidence` | Capped per §5 propagation. |
| `needs_human_confirmation` | Per §5 propagation. |
| `severity` | Suggested severity (§5 default, policy may adjust). |
| `suggested_fix_type` | e.g., rename-label / add-column / remove-column / reorder / change-visibility / change-required / hide-internal-field. |
| `frontend_override_forbidden` | Bool, from policy. |
| `backend_evidence_agrees_or_conflicts` | `agrees` / `conflicts` / `none`. |
| `project_memory_read` | Bool — confirms `project_memory.md` was read this run. |
| `module_memory_read` | Bool — confirms `module_memory.md` was read this run. |
| `handoff_ready_ai_fix_instruction` | Short instruction usable by the handoff/Codex prompt. |

If no findings: report states "no mismatches detected at current confidence" and still records `project_memory_read` / `module_memory_read`.

## 7. Handoff spec

`module-handoff` writes two files. Both must instruct fixing the **TARGET FRONTEND repo, NOT ERP QA Kit**, and perform **no auto-modification**.

### 7.1 `frontend_fix_handoff.md` (human-readable)

MUST contain:
- Exact **screen IDs** in scope.
- **Expected labels / columns / fields** (from source of truth: screen_spec then backend).
- **Actual FE findings** (current FE values, per mismatch).
- **Likely FE files/components** to edit (`detected_routes_components` + `source_files`).
- **Related API/backend evidence** (endpoints, DTO fields, `backend_evidence_agrees_or_conflicts`).
- **Policy reminder:** frontend override is **forbidden** where policy says so — FE must conform to truth, not redefine it.
- **Explicit confirmation** that `project_memory.md` + `module_memory.md` were read.
- **Validation steps after fix** (re-run `extract-frontend` → `compare-contract` → `module-report`; confirm finding cleared).

### 7.2 `codex_frontend_fix_prompt.md` (Codex prompt)

A ready-to-paste prompt for Codex that:
- States the **target is the TARGET FRONTEND repo**, never ERP QA Kit.
- Lists screen IDs, expected vs actual, likely files/components, related backend evidence.
- Embeds the **frontend-override-forbidden** policy reminder.
- States `project_memory.md` + `module_memory.md` were read.
- Includes post-fix **validation steps**.
- Forbids auto-modification of any ERP/backend code and of the QA Kit; humans approve before any change lands.

## 8. Confidence & provenance rules (restated)

| Source / extractor | Allowed confidence | needs_human_confirmation |
|---|---|---|
| Structured YAML/JSON, explicit | may be `high` | may be `false` |
| xlsx text / csv / markdown | default `medium` | default `true` |
| Static FE/BE heuristics (inferred mappings) | `medium`/`low` | `true` for inferred mappings |
| Static procedure SQL (client, unverified) | `medium`/`low` | `true` (intended-but-unverified) |
| OCR / image-derived / MarkItDown-from-image | `low`/`medium`, **never `high`** | **ALWAYS `true`** |
| Comparison finding | capped at lower of the two compared sides | `true` if either side low conf |

Rule: extraction **never** equals truth directly — it is normalized into contracts carrying `source` + `confidence` + `needs_human_confirmation`. Human confirmation (`needs_human_confirmation: false`) is preserved across re-runs (no-clobber). `xlsx_image_manifest_adapter` lists embedded images + positions only — **NO OCR**.

## 9. Policy/memory read requirement per command

Every command reads **project policy + project memory FIRST**; module commands additionally read **module policy + module memory**.

| Command | Reads project policy? | Reads project memory? | Reads module policy/memory? | If required policy missing |
|---|---|---|---|---|
| `policy-init` | if present | if present | n/a | Scaffolds it (this is the bootstrap command); exit `0`. |
| `module-init` | yes | yes | creates them | Project policy absent → exit `1` (run `policy-init` first). |
| `extract-spec` | yes (first) | yes (first) | yes (first) | Missing project/module policy → exit `1`. |
| `extract-frontend` | yes (first) | yes (first) | yes (first) | Missing project/module policy → exit `1`. |
| `extract-backend` | yes (first) | yes (first) | yes (first) | Missing project/module policy → exit `1`. |
| `extract-procedure` | yes (first) | yes (first) | yes (first) | Missing project/module policy → exit `1`. |
| `compare-contract` | yes (first) | yes (first) | yes (first) | Missing policy → exit `1`; honors precedence & `frontend_override_forbidden`. |
| `module-report` | yes (first) | yes (first) | yes (first) | Missing policy → exit `1`; records `*_memory_read` booleans. |
| `module-handoff` | yes (first) | yes (first) | yes (first) | Missing policy → exit `1`; confirms memory reads in output. |
| `module-audit` | yes (first) | yes (first) | yes (first) | Missing project policy → exit `1` before any step; reads `validation_order` + `defer_until_frontend_verified`. |

Policy/memory schemas and `validation_order` / `defer_until_frontend_verified` semantics are defined in `v0.2_PROJECT_POLICY_SPEC.md` and `v0.2_PROJECT_MEMORY_SPEC.md`.
