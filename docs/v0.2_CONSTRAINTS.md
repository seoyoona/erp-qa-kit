# ERP QA Kit v0.2 — CONSTRAINTS

> **Audience:** AI coding agents (Codex/Claude) and human maintainers implementing **v0.2**.
> **Status:** Authoritative source of truth for **v0.2-specific** hard limits.
> **Inherits:** All hard limits in [`../CONSTRAINTS.md`](../CONSTRAINTS.md) (v0.1) apply unchanged.
> **Sibling v0.2 docs (do not duplicate them here):** the v0.2 extractor spec, the v0.2 dependency/extras spec, the v0.2 policy/memory spec, and the v0.1 source-of-truth docs `../SPEC.md`, `../ACCEPTANCE_CRITERIA.md`, `../ARCHITECTURE.md`, `../ROADMAP.md`, `integrations.md`. Reference them by name; do not restate their full contents.

Every constraint below has a **stable ID** (`VC1.x`). IDs are stable across edits: never renumber or reuse an ID for a different meaning. Each is written to be **explicit and testable** — point to a check, a test, or an inspectable artifact that proves it holds. If you cannot prove it holds, treat it as **violated**.

Core principle (unchanged from v0.1, framing all of v0.2):

> **LLMs PROPOSE. Deterministic extractors/checks VERIFY. Humans APPROVE.** The target user is often a non-developer PM/QA. The toolkit **never** modifies target ERP code, **never** executes DB queries, and **only** writes under `qa-context/`.

## VC1. Purpose & Inheritance

- **VC1.1 — Inheritance.** v0.2 inherits **every** hard limit in `../CONSTRAINTS.md` (v0.1) without exception. Nothing in this document weakens, repeals, or grants an exception to a v0.1 constraint.
- **VC1.2 — Additive scope.** This document **adds** v0.2-specific limits on top of v0.1. It does not replace `../CONSTRAINTS.md`.
- **VC1.3 — Stricter-wins on conflict.** If a v0.2 limit and a v0.1 limit ever appear to conflict, the **stricter** (more restrictive, more human-gated, more containment-preserving) interpretation wins. When strictness is ambiguous, choose the reading that best protects: (a) non-destruction of target ERP, (b) write containment, (c) human approval, in that order.
- **VC1.4 — No new requirements.** Do not invent features, schemas, dependencies, repos, or integrations beyond what this doc and its named v0.2 siblings state. Record assumptions rather than expanding scope (consistent with `../INDEX.md`).

## VC2. Non-Destructive Constraints

Reinforces and extends v0.1 §3, §5, §11.

- **VC2.1 — No DB query execution, ever.** v0.2 must never open, configure, require, or use a database connection, and must never execute any SQL or stored procedure against any database. Procedure text may be **read as evidence only**; it is never run.
- **VC2.2 — No modification of target ERP code.** The toolkit must not modify, write, patch, refactor, delete, rename, or move any file in the target ERP source tree — **frontend, backend, OR stored procedures alike**. Target ERP and extracted module folders are **read-only input**.
- **VC2.3 — Single write zone (reuse v0.1 guard).** All writes land **only** under `qa-context/`. v0.2 must **reuse the existing v0.1 write-containment guard** — it must not introduce a second, weaker write path. Every new v0.2 write operation resolves its output path and asserts containment within `qa-context/`, raising and aborting otherwise.
- **VC2.4 — No new escape hatch.** No v0.2 command, extractor, adapter, or extra may open a file outside `qa-context/` in write/append mode. A test must prove an out-of-zone write attempt by a v0.2 code path is rejected.

## VC3. Source-of-Truth Constraints

Frontend is the **implementation under test**, not the truth.

- **VC3.1 — Frontend is not assumed correct.** The toolkit must never treat the frontend as authoritative. The frontend is the artifact being verified.
- **VC3.2 — Source of truth.** For **visible text, labels, columns, search filters, and form fields**, the source of truth is the **screen spec + backend evidence** — not the rendered frontend.
- **VC3.3 — Backend not fully trusted.** Backend is evidence, not absolute truth; client stored procedures may reflect intended logic but are **unverified**. Where screen spec and backend evidence disagree, surface the disagreement rather than silently picking one.
- **VC3.4 — `frontend_override_allowed=false` is strict.** When policy sets `frontend_override_allowed=false` (the pilot default), the toolkit must **never** resolve a conflict in the frontend's favor. It must **REPORT** the conflict for human resolution. Silently rewriting the source of truth to match the frontend is a violation.
- **VC3.5 — Priority order.** First priority is **frontend screen correctness**. Procedure / business-logic validation is **DEFERRED** until frontend mismatches are resolved; v0.2 must not block frontend checks on deferred procedure validation.

## VC4. Confidence / Extraction Constraints

Extends v0.1 §8.

- **VC4.1 — Extraction is never directly truth.** Raw extractor output must never be used directly as source of truth. It is **normalized into contracts** carrying the provenance trio (see VC6) before any downstream use.
- **VC4.2 — OCR / image-derived defaults.** Text derived from OCR or images defaults to **`low` or `medium`** confidence and **`needs_human_confirmation: true`**, and may **never** be assigned `high` confidence.
- **VC4.3 — Low-confidence spec is not perfect.** A screen spec whose extraction confidence is `low` must not be treated as perfect/authoritative; its low confidence propagates so downstream items remain flagged for human confirmation.
- **VC4.4 — Uncertain mapping flags itself.** Any uncertain interpretation or mapping (ambiguous column, unclear label, fuzzy match) must set **`needs_human_confirmation: true`**.

---

## VC5. Module-Scope Constraints

Pilot reality: 7 modules (1 verified), 3 repos incl. ORD, repos too big to fully clone.

- **VC5.1 — No full-repo analysis under `--module`.** When a `--module` scope is provided, the toolkit must scope its analysis to that module and must **not** perform a full-repo crawl/analysis.
- **VC5.2 — No mandatory full clone.** v0.2 must **not require** cloning all repositories to function. It must operate on partial, module-scoped inputs.
- **VC5.3 — Module-extracted folder is a first-class input.** The toolkit **must support** a module-specific extracted folder as input (the preferred pilot shape), not only full repo roots.
- **VC5.4 — Multi-repo path mapping comes from policy.** Mapping of modules to repositories/paths (incl. ORD and the other repos) is read from **policy**, not hardcoded and not inferred from a full-repo scan.

---

## VC6. Provenance Constraints

Consistent with the v0.1 validator (§8); applies to all new v0.2 inferred outputs.

- **VC6.1 — Trio on every inferred item.** Every inferred item — **contracts, mismatches, assumptions, inventory entries** — must carry `source`, `confidence`, and `needs_human_confirmation`.
- **VC6.2 — Closed confidence enum.** `confidence` ∈ `{high, medium, low}`; no other value is permitted. `needs_human_confirmation` is a boolean.
- **VC6.3 — Caution default.** Any item with `confidence` of `low` or `medium` implies `needs_human_confirmation: true`. Only `high` items may set it `false`, and a human may set it back to `true`. The pipeline must not silently flip it `true → false`.
- **VC6.4 — Validator parity.** v0.2 inferred items must pass the **same** v0.1 validation rules for the provenance trio (no relaxation, no new exempt item types).

## VC7. Dependency Constraints

Extends v0.1 §4, §9. No dependencies are added during planning.

- **VC7.1 — One new required dep.** **`openpyxl`** is the **only** new REQUIRED runtime dependency permitted in v0.2. All other required adapters use the **standard library + PyYAML** only.
- **VC7.2 — Optional extras degrade gracefully.** `pandas` and **Microsoft MarkItDown** are **OPTIONAL extras only**. When absent, the affected command must degrade gracefully (clear message, no crash) and the rest of the toolkit must still run.
- **VC7.3 — Docling / Unstructured are FUTURE-only.** **Docling** and **Unstructured** must **NOT** be added to dependencies and must **NOT** be imported anywhere in v0.2. They are documented as future-only (see `integrations.md`), implemented nowhere.
- **VC7.4 — No other new required deps.** No required dependency beyond `openpyxl` (plus v0.1's PyYAML and stdlib) may be added.
- **VC7.5 — No required network calls.** Running any v0.2 command must require **no** network access (reinforces v0.1 §4). No telemetry, no phone-home.

---

## VC8. Extractor Integrity Constraints

Extends v0.1 §7, §9. Reference the v0.2 extractor spec sibling for per-extractor detail.

- **VC8.1 — REQUIRED extractors must be real.** Any extractor marked **REQUIRED** in v0.2 must be: (a) actually implemented (not a placeholder folder), (b) **wired to its CLI command**, (c) **covered by tests or fixtures**, and (d) **documented**. Missing any of these four is a release failure.
- **VC8.2 — No placeholder pretending to be required.** A placeholder/empty/stub extractor must never be presented or wired as a working REQUIRED extractor.
- **VC8.3 — Optional/future are gated.** Optional and future extractors must be **clearly gated** (extra/feature flag) and must **never become accidentally required** — importing the toolkit's required path must not import them.
- **VC8.4 — Graceful absence.** When an optional extractor's dependency is missing, the toolkit reports it cleanly and continues; it does not hard-fail the pipeline.

---

## VC9. Policy / Memory Constraints

- **VC9.1 — Read before acting.** Every command/skill must read **project policy + project memory** (and, where applicable, **module policy + module memory**) **before** doing work.
- **VC9.2 — Missing `project_policy.yaml` halts or drafts.** If `project_policy.yaml` is missing, the command must either **stop** or **create a draft requiring human confirmation** (`needs_human_confirmation: true`). It must not proceed silently with invented policy.
- **VC9.3 — Cite memory in outputs.** Reports and handoffs must **state that policy/memory were read** (and which were missing/drafted).
- **VC9.4 — Module overrides project.** `module_memory` **refines/overrides** project memory when both apply; conflicts between layers are resolved by the more specific (module) layer.
- **VC9.5 — Conflicts reported, not resolved silently.** Frontend-vs-policy conflicts (and policy-vs-evidence conflicts) are **REPORTED**, not silently resolved. `frontend_override_allowed=false` is strictly enforced (see VC3.4).

## VC10. Backward-Compatibility Constraints

- **VC10.1 — v0.1 commands unchanged.** All v0.1 CLI commands (`init → intake → validate → generate-sql → report → handoff`) keep working with unchanged behavior and flags.
- **VC10.2 — File formats unchanged.** Existing v0.1 file formats (YAML schemas, `qa-context/` layout, report/handoff artifacts) remain readable and writable as before. No breaking format change.
- **VC10.3 — v0.1 tests pass unchanged.** The existing v0.1 test suite must continue to pass **without modification**.
- **VC10.4 — Purely additive.** v0.2 changes are **additive** (new commands, new optional adapters, new fields with safe defaults). Existing files in v0.1 are not modified by this planning effort and v0.2 must not require editing v0.1 behavior to function.

## VC11. Anti-Overengineering Constraints

Reinforces v0.1 §7.2 (OUT OF SCOPE).

- **VC11.1 — No dashboard.** No web/GUI dashboard in v0.2.
- **VC11.2 — No Playwright / browser automation** as a runtime dependency or wired feature.
- **VC11.3 — No API test execution.** No live API test running/generation engine.
- **VC11.4 — No model-based testing** (e.g., GraphWalker) implementation.
- **VC11.5 — No heavy third-party QA engine integration.** Such tools remain documentation-only in `integrations.md`.
- **VC11.6 — Keep the CLI simple.** v0.2 stays a simple, local, file-based CLI consistent with v0.1's shape.
