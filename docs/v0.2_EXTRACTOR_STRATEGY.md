## 1. Strategy & Principles

The v0.2 extractor layer exists to convert messy real-world inputs (spec spreadsheets, CSVs, markdown, JSON/YAML, frontend/backend source trees, stored-procedure SQL) into **normalized contracts** that downstream LLM skills and deterministic checks consume. The governing principles are non-negotiable:

1. **Deterministic adapters do the parsing — not an LLM.** File reading, cell traversal, route scanning, and SQL tokenizing are performed by deterministic Python code and/or established open-source libraries. We never rely on an LLM to "just read the whole file" and infer structure. The adapter layer is the boundary between bytes-on-disk and meaning.

2. **LLM skills only INTERPRET already-extracted content.** Skills receive normalized, structured extraction output and map/normalize it into the contract vocabulary (visible text, columns, fields, buttons, endpoints, params, etc.), attaching `source`, `confidence`, and `needs_human_confirmation`. Skills do not open raw files.

3. **Extraction is NEVER the source of truth.** Adapter output is *evidence*, not truth. It flows into a `*_contract.yaml` where every item carries provenance and a confirmation flag. Truth is established only by the comparison pipeline plus human approval (per `docs/v0.2_SPEC.md`). Per the SSOT rule, source of truth for visible text/labels/columns/filters/fields is **screen spec + backend evidence**, never the frontend.

4. **OCR / image-derived text is low-trust by default.** Any text that originates from an image (embedded picture, screenshot, OCR, image-to-markdown conversion) defaults to **low or medium** confidence and **`needs_human_confirmation: true` ALWAYS**. It may NEVER be promoted to `high`. v0.2 does not perform OCR at all in REQUIRED adapters — images are *manifested*, not read.

5. **Containment & provenance reuse.** Every adapter write goes through the v0.1 `assert_within_qa_context` guard (writes only under `qa-context/`), and every contract item reuses the v0.1 provenance constants in its `source` / `extraction_method` field. Adapters never modify target ERP code and never execute a database query.

6. **One required dependency.** The ONLY new REQUIRED runtime dependency in v0.2 is **openpyxl**. Everything else REQUIRED uses the Python stdlib (`csv`, `json`, plain text) plus the existing **PyYAML**. `pandas` and Microsoft **MarkItDown** are OPTIONAL import-guarded extras. **Docling** and **Unstructured** are FUTURE-only — documented, never imported, never added as dependencies. No dependencies are added during this planning step.

## 2. Adapter Registry Design

The registry is the single dispatch point that maps an input to the adapter(s) that can handle it. It lives under `erpqa/adapters/extractors/`.

### Interface (summary — full signatures in `docs/v0.2_ARCHITECTURE.md`)

Each adapter is a small object/class exposing at minimum:

- `name` — stable identifier (e.g. `xlsx_text_adapter`).
- `supported_extensions` — set of lowercased extensions it claims (e.g. `{".xlsx", ".xlsm"}`).
- `tier` — one of `REQUIRED` / `OPTIONAL` (FUTURE adapters are not objects in v0.2; see §6).
- `availability()` — returns whether the adapter's underlying library is importable and usable *right now*. REQUIRED adapters are always available (stdlib/openpyxl/PyYAML). OPTIONAL adapters return `False` when their extra is not installed.
- `extract(path, context) -> RawExtraction` — performs the deterministic parse and returns a raw, structured extraction (NOT yet a contract). Normalization into the `*_contract.yaml` happens in the command/skill layer, which attaches `source` / `confidence` / `needs_human_confirmation`.

### Dispatch & gating

Selection of an adapter for a given file is gated by **two independent conditions**, both of which must pass:

1. **Policy gate — `allowed_file_formats`.** Project/module policy (see `docs/v0.2_SPEC.md` / project-policy spec) declares which file formats are permitted for this project. An adapter whose extension is not in `allowed_file_formats` is skipped even if it exists. This lets a project lock itself to, say, `.xlsx + .md` only.
2. **Availability gate — `availability()`.** An adapter only handles a file if its library is importable. REQUIRED adapters always pass. OPTIONAL adapters pass only when their extra is installed.

Resolution: `extension → candidate adapters → filter by policy allowed_file_formats → filter by availability() → choose`. When more than one available adapter claims an extension (e.g. `.xlsx` served by `xlsx_text_adapter` plus optional `pandas`), tier/priority and policy decide; REQUIRED adapters are the default and OPTIONAL adapters only augment, never silently replace.

### Registration rules

- **REQUIRED adapters** register unconditionally at import time. They are always present.
- **OPTIONAL adapters** register **only when their underlying library is importable**. Registration is import-guarded: if `import pandas` / `import markitdown` fails, the adapter is not added to the registry at all, so it can never be accidentally selected. Absence is graceful — the command falls back to the REQUIRED adapter for that extension (or reports "no available adapter" cleanly).
- **FUTURE adapters (Docling, Unstructured)** are **NOT registered, not imported, and not present as objects** in v0.2. They exist only as documentation in §6. There is no code path that can reach them.

## 4. REQUIRED v0.2 Adapters (implement now)

All eight below are **implement-now = YES**: real, wired to their CLI command, covered by tests/fixtures, and documented. None may ship as a placeholder.

### 4.1 `xlsx_text_adapter`

- **Underlying library:** openpyxl. Docs: https://openpyxl.readthedocs.io/ ; GitHub: https://foss.heptapod.net/openpyxl/openpyxl
- **Supported extensions:** `.xlsx`, `.xlsm`.
- **Installation / dependency approach:** openpyxl is the single new REQUIRED runtime dependency, added to `pyproject` `[project.dependencies]`. Always available; `availability()` returns `True`.
- **CLI integration:** invoked by `erpqa extract-spec <project_path> --module <NAME>` over declared spec files.
- **Output format (raw → normalized):** raw extraction = per-sheet structure: sheet names, cell values with addresses, merged-cell ranges (resolved to their anchor value + spanned range), row/column grouping, and detected header rows. The skill normalizes this into `screen_contract.yaml` items (screen, visible text, columns, fields, buttons) per the schema in `docs/v0.2_SPEC.md`.
- **Normalized contract target:** `screen_contract.yaml`.
- **Confidence rules:** default **medium**; **`needs_human_confirmation: true`** (spreadsheet layout is human-authored and ambiguous; cell→meaning mapping is inferred).
- **Failure modes:** corrupt/non-OOXML file → fail clean (command exit `1`, no partial contract); password-protected/unsupported sub-format → unsupported, reported; empty workbook / all-blank sheets → empty extraction, item-less contract noting "no content"; ambiguous header/merged-cell mapping → emit item with `needs_human_confirmation: true` and a note, never guess silently.
- **Tests / fixtures:** small `.xlsx` with merged cells + multi-sheet; an `.xlsm`; a corrupt file; an empty workbook. Assert sheet/cell/merged-range extraction and that all items default to medium + needs_human_confirmation true.
- **Implement-now:** **YES.**

### 4.2 `xlsx_image_manifest_adapter`

- **Underlying library:** openpyxl. Docs: https://openpyxl.readthedocs.io/ ; GitHub: https://foss.heptapod.net/openpyxl/openpyxl
- **Supported extensions:** `.xlsx`, `.xlsm` that contain embedded images.
- **Installation / dependency approach:** uses the already-required openpyxl; always available.
- **CLI integration:** invoked by `erpqa extract-spec` alongside `xlsx_text_adapter` when the workbook embeds images.
- **Output format (raw → normalized):** produces a **MANIFEST of images** — one entry per embedded image with: sheet name, anchor cell, position/extent, image index, and (where available) original filename/dimensions. **It does NOT OCR and does NOT read pixels into text.** Normalization records each image as a contract item flagged for human attention (e.g. "screen-spec content may live inside this image; OCR deferred").
- **Normalized contract target:** `screen_contract.yaml`.
- **Confidence rules:** **low or medium**, and **`needs_human_confirmation: true` ALWAYS** (no exceptions). OCR is explicitly **deferred to a future tier** (see §6). The manifest tells a human *where* image content is, not *what it says*.
- **Failure modes:** corrupt file → clean fail; workbook with no images → empty manifest (valid, zero image items); unreadable/embedded-object-but-not-image → recorded as "image present, not extractable", needs_human_confirmation true.
- **Tests / fixtures:** an `.xlsx` with one or more embedded images anchored to cells; a workbook with zero images (expect empty manifest). Assert each image yields a manifest entry with sheet/anchor/position and that NO image text is produced and confirmation is always true.
- **Implement-now:** **YES.**

### 4.3 `csv_adapter`

- **Underlying library:** Python stdlib `csv` (no external dep).
- **Supported extensions:** `.csv`.
- **Installation / dependency approach:** stdlib only; always available.
- **CLI integration:** invoked by `erpqa extract-spec` over declared `.csv` spec files.
- **Output format (raw → normalized):** raw = header row + typed-as-text rows with delimiter/dialect sniffing; normalized into tabular `screen_contract.yaml` items (columns from header; row values as candidate field/visible-text evidence).
- **Normalized contract target:** `screen_contract.yaml` (tabular).
- **Confidence rules:** default **medium**; **`needs_human_confirmation: true`** (header→column-meaning is inferred).
- **Failure modes:** malformed/ragged rows → record rows that parse, flag ragged rows with a note; wrong/ambiguous delimiter → dialect sniff, fall back to comma, note low confidence; empty file / header-only → empty or columns-only contract; encoding errors → attempt utf-8 then report.
- **Tests / fixtures:** clean comma CSV; semicolon/tab-delimited CSV; ragged-row CSV; empty CSV. Assert header→columns and medium + needs_human_confirmation true.
- **Implement-now:** **YES.**

### 4.4 `markdown_adapter`

- **Underlying library:** Python stdlib (text processing; no external markdown lib required).
- **Supported extensions:** `.md`.
- **Installation / dependency approach:** stdlib only; always available.
- **CLI integration:** invoked by `erpqa extract-spec` over declared `.md` spec files.
- **Output format (raw → normalized):** raw = structural parse of headings, lists, tables (GFM pipe tables), and paragraphs; normalized into `screen_contract.yaml` (headings → screen/section names; tables → columns; list items / labeled lines → fields/buttons/visible text).
- **Normalized contract target:** `screen_contract.yaml`.
- **Confidence rules:** default **medium**; **`needs_human_confirmation: true`** (markdown structure is loose; mapping is inferred).
- **Failure modes:** malformed tables → extract what parses, flag; binary/non-text content → reject as unparseable; empty file → empty contract; deeply nested/ambiguous structure → emit with confirmation true.
- **Tests / fixtures:** markdown with headings + a GFM table + lists; a malformed-table markdown; an empty file. Assert headings/tables/lists extraction and medium + needs_human_confirmation true.
- **Implement-now:** **YES.**

### 4.5 `json_yaml_adapter`

- **Underlying library:** Python stdlib `json` + PyYAML (existing dependency).
- **Supported extensions:** `.json`, `.yaml`, `.yml`.
- **Installation / dependency approach:** stdlib + already-present PyYAML; always available.
- **CLI integration:** invoked by `erpqa extract-spec` over declared structured spec files.
- **Output format (raw → normalized):** raw = the fully parsed object tree; normalized into `screen_contract.yaml` by mapping known/labeled keys (e.g. `screen`, `columns`, `fields`, `buttons`, `visible_text`) directly to contract items.
- **Normalized contract target:** `screen_contract.yaml`.
- **Confidence rules:** default **medium** with **`needs_human_confirmation: true`** for loosely-structured input; **MAY be `high`** (and `needs_human_confirmation: false`) **only when the input is explicitly labeled/structured** to the expected contract vocabulary (i.e. an authored spec that already speaks the contract schema). High is the sole REQUIRED-tier case where confirmation can be false — and only because the structure is explicit, not inferred. Image-derived content is never high (not applicable here — this adapter reads structured text, not images).
- **Failure modes:** invalid JSON/YAML → unparseable, clean fail; YAML with unsafe tags → safe-load only; ambiguous/unrecognized schema → fall back to medium + needs_human_confirmation true (never auto-promote to high); empty file → empty contract.
- **Tests / fixtures:** a JSON spec using the explicit contract vocabulary (expect high / confirmation false); a loosely-structured YAML (expect medium / confirmation true); invalid JSON; invalid YAML. Assert the high-vs-medium decision is driven solely by explicit labeling.
- **Implement-now:** **YES.**

### 4.6 `frontend_static_adapter`

- **Underlying library:** deterministic static scan using the Python stdlib (text/regex/structural pattern scanning over frontend source). NO LLM, NO browser, NO execution.
- **Supported extensions:** frontend source files in the module-scoped frontend folder (e.g. component/template/markup/script files declared in the module manifest). Scope is module-bound — no full-repo crawl (per `docs/v0.2_CONSTRAINTS.md`).
- **Installation / dependency approach:** stdlib only; always available.
- **CLI integration:** invoked by `erpqa extract-frontend <project_path> --module <NAME>`.
- **Output format (raw → normalized):** raw = detected **routes, components, visible text/labels, table columns, form fields, buttons**, with file + line provenance. Normalized into `frontend_contract.yaml` per `docs/v0.2_SPEC.md`.
- **Normalized contract target:** `frontend_contract.yaml`.
- **Confidence rules:** **medium for directly-extracted literals; low for inferred mappings**; **`needs_human_confirmation: true` for any inferred mapping** (e.g. binding a column header to a data field). Remember the SSOT rule: the frontend is the implementation under test, not the source of truth — its contract is evidence, never authoritative.
- **Failure modes:** unparseable/exotic syntax → extract what is statically detectable, flag the rest; no frontend source declared → command exit `1`; empty module folder → empty contract; dynamic/computed text that can't be resolved statically → record as "dynamic, unresolved", needs_human_confirmation true.
- **Tests / fixtures:** a small fixture frontend module with a route, a component containing a table (columns), a form (fields), buttons, and static labels; a module with dynamic/computed text. Assert routes/components/columns/fields/buttons extraction with provenance and correct confidence split.
- **Implement-now:** **YES.**

### 4.7 `backend_static_adapter`

- **Underlying library:** deterministic static scan using the Python stdlib over backend source. NO LLM, NO execution, NO DB connection.
- **Supported extensions:** backend source files in the module-scoped backend folder declared in the module manifest. Module-bound; no full-repo crawl.
- **Installation / dependency approach:** stdlib only; always available.
- **CLI integration:** invoked by `erpqa extract-backend <project_path> --module <NAME>`.
- **Output format (raw → normalized):** raw = detected **endpoints, DTO/model fields, service & repository file references, stored-procedure call sites, and field-mapping evidence** (e.g. DTO field ↔ column name), each with file + line provenance. Normalized into `backend_contract.yaml`.
- **Normalized contract target:** `backend_contract.yaml`.
- **Confidence rules:** **medium for directly-detected declarations; low for inferred field mappings**; **`needs_human_confirmation: true`** for inferred mappings. Backend evidence is part of the source-of-truth basis (with screen spec), so accuracy and clear provenance matter especially here.
- **Failure modes:** unparseable/multi-language backend → extract detectable declarations, flag gaps; no backend source declared → exit `1`; empty folder → empty contract; ambiguous DTO↔column mapping → low + needs_human_confirmation true; any code path that would *connect to a DB* is forbidden by design (safety violation, exit `3`).
- **Tests / fixtures:** a fixture backend module with an endpoint, a DTO with fields, a service+repository, a stored-procedure call site, and a clear field mapping; plus an ambiguous-mapping case. Assert endpoints/DTO fields/files/proc-calls/mapping evidence extraction with provenance and confidence split.
- **Implement-now:** **YES.**

### 4.8 `procedure_sql_adapter`

- **Underlying library:** deterministic SQL **text parse** using the Python stdlib over stored-procedure files. **NO execution. NO DB connection — ever.**
- **Supported extensions:** stored-procedure SQL files (e.g. `.sql`) in the module-scoped procedure folder declared in the manifest.
- **Installation / dependency approach:** stdlib only; always available.
- **CLI integration:** invoked by `erpqa extract-procedure <project_path> --module <NAME>`. Note (per `docs/v0.2_SPEC.md`): extraction is cheap/safe and runs now, but downstream procedure *validation* is deferred until frontend mismatches are resolved (`defer_until_frontend_verified`).
- **Output format (raw → normalized):** raw = per-procedure **name, parameters (name/direction/type where detectable), result/output columns, and tables touched** (FROM/JOIN/INSERT/UPDATE targets where statically detectable), with provenance. Normalized into `procedure_contract.yaml`.
- **Normalized contract target:** `procedure_contract.yaml`.
- **Confidence rules:** **medium for clearly-declared params/name; low for inferred result columns / tables touched**; **`needs_human_confirmation: true`** (SQL static parse is heuristic; dynamic SQL and complex queries are not fully resolvable).
- **Failure modes:** unparseable SQL / no procedure source → exit `1`; dynamic SQL (string-built) → record "dynamic, unresolved", needs_human_confirmation true; multiple dialects → best-effort tokenizing, flag uncertainty; **any DB-connection attempt is a `3` safety violation by design** and must be structurally impossible.
- **Tests / fixtures:** a fixture procedure file with named params, a result SELECT, and tables in FROM/JOIN; a dynamic-SQL procedure; an unparseable file. Assert name/params/result-columns/tables extraction, confidence split, and that no execution/connection code exists.
- **Implement-now:** **YES.**

## 5. OPTIONAL v0.2 Adapters (guarded; must NOT become required)

Both are **implement-now = OPTIONAL / guarded**. They register only when their library is importable, augment (never replace) the REQUIRED adapters, and the system behaves identically when they are absent. **Neither may ever become a required dependency.**

### 5.1 pandas (augments tabular extraction)

- **Official links (verbatim):** Docs: https://pandas.pydata.org/docs/ ; GitHub: https://github.com/pandas-dev/pandas
- **What it adds:** richer/faster tabular parsing for large CSVs and multi-sheet spreadsheets — robust dtype handling, large-file performance, and convenient sheet/dataframe traversal — as an *augmentation* of `csv_adapter` and the xlsx text path.
- **Dependency-extra approach:** declared in `pyproject` `[project.optional-dependencies]` (e.g. an extra such as `pandas`). Import-guarded; not in the base install.
- **Graceful-absence behavior:** if `import pandas` fails, the pandas augmentation never registers and the REQUIRED `csv_adapter` / `xlsx_text_adapter` handle the file exactly as before. No feature is lost, no error is raised for absence.
- **Confidence rules:** unchanged from the REQUIRED tabular adapters — default **medium**, **`needs_human_confirmation: true`** (pandas changes *how* we read tables, not *how much we trust* the inferred mapping).
- **Failure modes:** same as the REQUIRED tabular adapters (corrupt/empty/ragged/ambiguous). On any pandas-specific error, fall back to the REQUIRED stdlib path.
- **Tests / fixtures:** tests must run and pass with pandas absent (skip/fallback path) and, when pandas is installed, assert parity of extraction output with the stdlib path.
- **Must NOT become required:** explicitly OPTIONAL. The base install and all REQUIRED behavior must work with pandas uninstalled.

### 5.2 `markitdown_adapter` (Microsoft MarkItDown)

- **Official links (verbatim):** GitHub: https://github.com/microsoft/markitdown
- **What it adds:** conversion of additional document formats (e.g. office/PDF-style documents and images) into markdown text, which then feeds the markdown normalization path into `screen_contract.yaml`. Broadens the set of spec inputs we can read without adding heavyweight FUTURE tooling.
- **Dependency-extra approach:** declared in `pyproject` `[project.optional-dependencies]` (e.g. an extra such as `markitdown`). Import-guarded; not in the base install.
- **CLI integration:** when present and permitted by policy `allowed_file_formats`, it participates in `erpqa extract-spec` for the extra formats it claims.
- **Graceful-absence behavior:** if `import markitdown` fails, the adapter does not register; those extra formats are simply unsupported (clean "no available adapter" message), and all REQUIRED adapters are unaffected.
- **Confidence rules:** default **medium** with **`needs_human_confirmation: true`**; and critically, **any image-derived content from MarkItDown is low/medium and `needs_human_confirmation: true`** — it may NEVER be `high` (it is image/OCR-class evidence; see the hard rule in §7).
- **Failure modes:** conversion error / unsupported document → clean fail, fall through to REQUIRED adapters where applicable; empty conversion → empty contribution; image-only document → low confidence, confirmation true.
- **Tests / fixtures:** tests must pass with MarkItDown absent (skip path); when installed, assert a supported document converts and that image-derived items are capped at medium with confirmation true.
- **Must NOT become required:** explicitly OPTIONAL. No REQUIRED behavior may depend on MarkItDown being installed.

## 7. Confidence & Provenance Rules Table

Every contract item carries `source` / `extraction_method` (reusing v0.1 provenance constants), `confidence`, and `needs_human_confirmation`.

| Source type | Default confidence | needs_human_confirmation |
|---|---|---|
| Explicitly-labeled structured spec (JSON/YAML speaking the contract vocabulary) | **high** (allowed) | **false** (allowed) |
| XLSX/XLSM cells & merged cells (`xlsx_text_adapter`) | medium | true |
| XLSX/XLSM embedded image **manifest** (`xlsx_image_manifest_adapter`, no OCR) | low / medium | **true (always)** |
| CSV tabular (`csv_adapter`) | medium | true |
| Markdown (`markdown_adapter`) | medium | true |
| Frontend static — direct literals | medium | true |
| Frontend static — inferred mappings | low | true |
| Backend static — direct declarations | medium | true |
| Backend static — inferred field mappings | low | true |
| Procedure SQL — declared name/params | medium | true |
| Procedure SQL — inferred result columns / tables touched | low | true |
| pandas-augmented tabular | medium | true |
| MarkItDown text (non-image) | medium | true |
| **Any OCR / image-derived text (MarkItDown image, future Docling/Unstructured OCR)** | **low or medium (capped)** | **true (always)** |

**HARD RULE:** OCR / image-derived content **never exceeds `medium`** and **always carries `needs_human_confirmation: true`**. It may NEVER be `high`. The only path to `high` (with `needs_human_confirmation: false`) is an explicitly-labeled structured text spec — never an image, never an inference.

## 8. Anti-Placeholder Rule (binding restatement)

Every adapter marked **REQUIRED v0.2** (the eight in §4) must be:

1. **Real** — a working deterministic implementation, not a stub or `NotImplementedError`.
2. **Wired** — registered in the registry and invoked by its CLI command (`extract-spec`, `extract-frontend`, `extract-backend`, `extract-procedure`) per `docs/v0.2_SPEC.md`.
3. **Tested with fixtures** — concrete fixture files (including corrupt/empty/ambiguous cases) and assertions on extraction output and on the confidence / `needs_human_confirmation` defaults.
4. **Documented** — in this strategy and in user-facing docs/help.

OPTIONAL adapters (§5) must be import-guarded and fully degrade-gracefully, and tests must pass with them absent. FUTURE adapters (§6) must remain documentation only — no import, no registry entry, no dependency. No REQUIRED behavior may ever depend on an OPTIONAL or FUTURE library.
