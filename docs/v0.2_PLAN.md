# ERP QA Kit v0.2 — Master Plan

> **Status:** Planning only. This document defines *what* v0.2 is and *how it should be built*. No code is written and no v0.1 file is modified during planning. This is the authoritative overview; detailed specifications live in the sibling documents referenced throughout.

---

## 1. Purpose & Relationship to v0.1

ERP QA Kit **v0.1.0** already shipped as a local Python CLI (`erpqa`) that performs: `qa-context init`, intake, YAML validation, SELECT-only SQL safety, `DB_ASSERTION → SQL` generation, Markdown reports, feedback-to-fix handoff, a demo, and 19 passing tests. Its stack is intentionally minimal: Python `>=3.11`, a **single** runtime dependency (PyYAML), an `argparse` CLI where each subcommand takes a positional `project_path`, a write-containment guard (`erpqa/core/paths.py: assert_within_qa_context`) that refuses any write outside `qa-context/`, and provenance constants in `erpqa/core/constants.py`.

**v0.2 extends v0.1; it never breaks it.**

- v0.2 **MUST NOT** change v0.1 behavior or break any of its 19 tests.
- v0.2 reuses, never rewrites, the v0.1 foundations: the write-containment guard, provenance constants, the `argparse` CLI conventions, and the flat-hyphenated subcommand style (e.g. v0.1's `generate-sql`).
- v0.2 preserves the **core principle**: *LLMs PROPOSE; deterministic checks/extractors VERIFY; humans APPROVE.*
- v0.2 preserves the **hard guarantees**: the toolkit NEVER modifies target ERP code, NEVER executes DB queries, and only ever writes under `qa-context/` (plus emits handoff docs that humans/Codex act on).

The target user remains a non-developer PM/QA lead, so v0.2 must stay simple to run.

## 2. v0.2 Objective & the 15 Focus Areas

**Objective:** add project-specific QA policy + persistent project/module memory + module-level screen-contract validation + executable file-extractor adapters, so the kit is usable across **many** ERP projects and runnable on a small **module-scoped extracted folder** (not the whole repo).

This is driven by the real pilot: an ERP with **7 modules** (only 1 verified so far), **3 repositories** (incl. ORD), repos too large to clone/analyze wholesale → **module-level extracted folders preferred**. **Frontend override is strictly forbidden.** Backend implementation is not fully trusted; client stored procedures may be closer to intended business logic but are integration-unverified. The **first validation priority is frontend screen correctness**; procedure/business-logic validation is **deferred** until frontend screen-contract mismatches are resolved. The **screen spec + backend evidence are the source of truth** for visible text, labels, columns, search filters, and form fields; the **frontend is the implementation to validate, not the source of truth.**

### The 15 Focus Areas

| # | Focus Area | One-line intent |
|---|------------|-----------------|
| 1 | Project policy | `project_policy.yaml` — machine-readable per-project QA rules (e.g. `frontend_override_allowed: false`, `validation_order`, `defer_until_frontend_verified`). |
| 2 | Project memory | `project_memory.md` — PM-readable durable knowledge; cited by every report/handoff. |
| 3 | Project assumptions | `project_assumptions.yaml` — recorded, human-confirmable assumptions. |
| 4 | Module policy | `modules/{MODULE}/module_policy.yaml` — refines/overrides project policy per module. |
| 5 | Module memory | `modules/{MODULE}/module_memory.md` — refines/overrides project memory per module. |
| 6 | Module audit | One-shot orchestrated run for a single module honoring policy ordering and defer rules. |
| 7 | Multi-format spec extraction | Read xlsx/csv/markdown/json/yaml specs into a normalized form with confidence. |
| 8 | `screen_contract.yaml` | The normalized source-of-truth contract (screen spec + backend evidence). |
| 9 | Frontend contract | `frontend_contract.yaml` — what the frontend actually implements (to be validated). |
| 10 | Backend contract | `backend_contract.yaml` — backend-derived evidence (medium/low confidence). |
| 11 | Procedure contract | `procedure_contract.yaml` — SQL-procedure-derived evidence (deferred priority). |
| 12 | Screen-vs-frontend comparison | Deterministic comparison of frontend against source of truth. |
| 13 | Mismatch report | PM-readable Markdown reports enumerating every detected mismatch. |
| 14 | Frontend fix handoff | `frontend_fix_handoff.md` + `codex_frontend_fix_prompt.md` for downstream fixing. |
| 15 | Extractor registry + optional/future integrations | Registry wiring required adapters; documenting optional (pandas, MarkItDown) and future (Docling, Unstructured). |

## 4. CLI Decision

**Recommendation: implement BOTH granular subcommands AND an orchestrated `module-audit`.**

**Rationale.**
- **Granular subcommands** let a developer/QA lead control or re-run a single stage (e.g. only re-extract the frontend after a code change, or re-run comparison after fixing a contract) without redoing everything.
- The **`module-audit` orchestrator** gives the non-developer PM a one-shot command that runs the stages in the policy's `validation_order` and honors `defer_until_frontend_verified` (so procedure validation is skipped until frontend mismatches are resolved).
- Both share the same engine, so there is no duplicated logic — the orchestrator simply sequences the granular stages.

**Conventions.** Flat, hyphenated subcommands consistent with v0.1's `generate-sql`. Each new command takes positional `project_path` plus `--module <NAME>`, **except `policy-init`** which is project-level (no module).

| Command | Level | Purpose |
|---------|-------|---------|
| `policy-init` | Project | Create/refresh `project_policy.yaml`, `project_memory.md`, `project_assumptions.yaml` (draft requires human confirmation). |
| `module-init` | Module | Scaffold `qa-context/modules/{MODULE}/` and `module_manifest.yaml`, `module_policy.yaml`, `module_memory.md`. |
| `extract-spec` | Module | Multi-format spec → normalized `screen_contract.yaml` (source of truth). |
| `extract-frontend` | Module | Static frontend scan → `frontend_contract.yaml`. |
| `extract-backend` | Module | Static backend scan → `backend_contract.yaml`. |
| `extract-procedure` | Module | SQL procedure text parse → `procedure_contract.yaml` (no execution). |
| `compare-contract` | Module | Frontend vs source-of-truth comparison → mismatch data. |
| `module-report` | Module | Render PM-readable mismatch/contract reports. |
| `module-handoff` | Module | Emit `frontend_fix_handoff.md` + `codex_frontend_fix_prompt.md`. |
| `module-audit` | Module | **Orchestrator** — run the above in `validation_order`, honoring `defer_until_frontend_verified`. |

## 5. Extractor Split (summary)

> Full design, signatures, normalization rules, and confidence mapping are in **`docs/v0.2_EXTRACTOR_STRATEGY.md`**. This is the summary only.

| Tier | Adapter | Backing tech | Default confidence |
|------|---------|--------------|--------------------|
| **REQUIRED** | `xlsx_text_adapter` | openpyxl | medium, needs confirm |
| **REQUIRED** | `xlsx_image_manifest_adapter` | openpyxl (lists embedded images + positions; **no OCR**) | low/medium, needs confirm |
| **REQUIRED** | `csv_adapter` | stdlib `csv` | medium, needs confirm |
| **REQUIRED** | `markdown_adapter` | stdlib text | medium, needs confirm |
| **REQUIRED** | `json_yaml_adapter` | stdlib `json` + PyYAML | high if explicitly machine-readable |
| **REQUIRED** | `frontend_static_adapter` | deterministic static scan | medium/low, needs confirm for inferred mappings |
| **REQUIRED** | `backend_static_adapter` | deterministic static scan | medium/low, needs confirm |
| **REQUIRED** | `procedure_sql_adapter` | deterministic SQL text parse (**no execution**) | medium/low, needs confirm |
| **OPTIONAL** | `pandas` (alt CSV/tabular) | pandas (extra) | as csv; graceful absence |
| **OPTIONAL** | `markitdown_adapter` | Microsoft MarkItDown (extra; Office/PDF/HTML/image → Markdown) | image-derived → low/at-most-medium, **always** needs confirm |
| **FUTURE** | Docling, Unstructured | — | document only; do not implement |

**Confidence rules (locked):** extraction never becomes source of truth directly — it is normalized into contracts carrying `source` / `confidence` / `needs_human_confirmation`. Explicitly-labeled machine-readable spec (yaml/json) → may be high. Tabular/text (xlsx text, csv, markdown) → default **medium**, `needs_human_confirmation: true` until confirmed. Static heuristic frontend/backend/procedure → **medium/low**, needs confirm for inferred mappings. OCR / image-derived / MarkItDown-from-image → **low (at most medium)** and `needs_human_confirmation` ALWAYS true; **NEVER high**.

## 7. Policy & Memory Behavior (summary)

> Full schemas/behavior in **`docs/v0.2_PROJECT_POLICY_SPEC.md`** and **`docs/v0.2_PROJECT_MEMORY_SPEC.md`**.

- **Every** command/skill reads project policy + memory (and module policy + memory) **BEFORE** doing work.
- If `project_policy.yaml` is missing → **stop**, OR create a **draft policy that requires human confirmation** (never silently proceed with defaults).
- Reports/handoffs **MUST cite** that `project_memory.md` (and `module_memory.md` if present) were read.
- `module_memory` / `module_policy` **refine or override** the project-level equivalents.
- Any **frontend-vs-policy conflict is REPORTED, never silently resolved.** When `frontend_override_allowed: false`, the frontend must not override screen spec/backend.
- Any uncertain interpretation → `needs_human_confirmation: true`.

## 9. Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| Low-confidence OCR / image-derived data treated as truth | Hard cap: image-derived → low (max medium), `needs_human_confirmation` always true, NEVER high; no OCR in required tier. |
| Heuristic frontend/backend parsing false positives | Static adapters mark inferred mappings as medium/low + needs confirm; comparison **reports** conflicts rather than auto-resolving. |
| Multi-repo / multi-module path mapping confusion | Module-scoped extracted folders + explicit `--module`; `module_manifest.yaml` records source paths; no full-repo assumptions. |
| Frontend silently overriding source of truth | `frontend_override_allowed: false` enforced; any override-shaped finding is a reported mismatch. |
| Procedure work distracting from frontend priority | `defer_until_frontend_verified` honored by `module-audit`; procedure validation deferred (extraction still allowed, low priority). |
| Scope creep (dashboards, browser automation, heavy engines) | Anti-overengineering rules (§10); NOT-in-scope table (§3.2). |
| Optional deps becoming accidentally required | Lazy imports + graceful degradation; tests confirm kit runs with only PyYAML + openpyxl. |
| Breaking v0.1 | No edits to v0.1 files; v0.1's 19 tests must remain green as a gate. |

## 10. Anti-Overengineering Rules

- No DB execution. No target-code modification.
- No full-repo analysis when a module is provided; do not require all repos to be cloned.
- Do **not** build dashboards, Playwright, model-based testing, or heavy third-party QA engines.
- Keep the CLI simple (flat hyphenated subcommands; one orchestrator).
- Required extractors must be **REAL + TESTED**, not placeholder folders.
- Optional/future extractors must **not** become accidentally required.
- Exactly one new required dependency (openpyxl); nothing else.

## 11. Definition of "v0.2 Done"

The authoritative completion checklist lives in **`docs/v0.2_ACCEPTANCE_CRITERIA.md`**. At a glance, v0.2 is done when:

- [ ] v0.1's 19 tests still pass; no v0.1 file modified.
- [ ] Policy/memory/assumptions files + module policy/memory implemented; every command reads them first; reports/handoffs cite memory reads.
- [ ] All 8 REQUIRED extractors implemented, registered, wired to the CLI, fixtured, tested, documented (no placeholders).
- [ ] All four contracts produced with source/confidence/`needs_human_confirmation`; confidence rules enforced.
- [ ] Comparison engine detects the full mismatch taxonomy; conflicts reported, never auto-resolved.
- [ ] Four reports + two handoff docs generated for a module.
- [ ] Both granular subcommands and `module-audit` work; `module-audit` honors `validation_order` + `defer_until_frontend_verified`.
- [ ] openpyxl is the only new required dep; kit runs gracefully without pandas/MarkItDown.
- [ ] Module-scoped demo + fixtures run end-to-end.

→ See **`docs/v0.2_ACCEPTANCE_CRITERIA.md`** for the precise criteria.

To exercise v0.2 against a real extracted module folder safely (no DB, no code changes, module-scoped only), follow **`docs/v0.2_DRY_RUN_GUIDE.md`**.
