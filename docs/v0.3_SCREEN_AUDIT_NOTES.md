# v0.3 (WIP) вҖ” Screen-Scoped Audit

Status: **prototype, validated on the real pipe-manufacturing ERP (module PDT)**.
This is additive to v0.2 and does not change v0.1/v0.2 behavior. It exists because the
v0.2 module-level extractor could not handle real screen-spec xlsx (formatted documents,
not clean tables) and modelled one screen per module вҖ” see the findings below.

## Command

```
erpqa screen-audit     <project_path> --module <MOD> --screen <SCREEN_ID> [--labels]
erpqa screen-audit-all <project_path> --module <MOD> [--labels]
```

`screen-audit-all` discovers every id-coded screen from the spec filenames, audits each, and
writes a consolidated `screens/_summary.{md,yaml}` вҖ” the human-facing deliverable. The summary
leads with the **trustworthy** CLEAN/ISSUES count (heuristic `sp-fallback` and OCR `--labels`
low-confidence findings excluded) and a "мӮ¬лһҢ мҡ°м„  кІҖнҶ " list of the high-confidence findings.
Run without `--labels` for the deterministic core; add `--labels` to also OCR screen labels
(every screen then has low-confidence label candidates, so the raw CLEAN drops to 0 вҖ” read the
trustworthy line instead).

Resolves three sources for one screen and compares them:
- **spec** вҖ” the screen's xlsx нҷ”л©ҙIO sheet. The reliable field source in this ERP is the
  embedded stored-procedure examples (`exec ..._IU @param=` for save, `..._S` for query).
- **backend** вҖ” the screen's module; pydantic `serialization_alias`/`validation_alias` and the
  service param dicts bridge SP params (`iInQty`) вҶ” clean keys (`in_qty`) вҶ” DB cols (`INQTY`).
- **frontend** вҖ” the feature file cluster (model + hooks), reached by following imports.

Output: `qa-context/modules/<MOD>/screens/<SCREEN_ID>/screen_audit{.yaml,_report.md}`.

## What is trustworthy (verified true positives)

- **Save-field comparison** (л“ұлЎқ screens): spec save-SP params vs frontend model fields.
  Verified on `PDT-OSC-001M` (found `pg_no`, `add_emp_no` missing in FE), `PDT-OSC-002M`,
  `PDT-OSC-004M`. Server-populated fields (audit/document-no) are flagged MINOR +
  `needs_human_confirmation: true`.
- **CLEAN detection**: when filters + columns fully match, the screen is reported consistent
  (e.g. `PDT-OSC-003M`, several `MG` screens).
- **Honesty guards**:
  - If the frontend model is not resolved (e.g. shared `src/models/` missing from the slice),
    emit ONE `FrontendModelUnresolved` notice instead of dozens of false "missing".
  - `BackendColumnNotInFrontend` is **info only**, never a finding вҖ” backend is *evidence*,
    not source of truth, and grids legitimately show a subset of returned columns.
  - Dynamic pivot columns (`specNN`/`dayNN`) are excluded from per-column comparison.

## Resolution precision (was the main blocker вҖ” addressed)

Backend binding now resolves in three tiers, each recorded in the audit output under
`resolution.method` (+ `resolution.confidence`):

1. **`explicit`** вҖ” a maintained `screen_bindings` entry in `module_manifest.yaml` pins the
   screen 1:1 to a backend dir (and optionally a frontend feature dir). Deterministic; the
   way to resolve the ambiguous м§„н–үнҳ„нҷ©/PRG group. Shape:
   ```yaml
   screen_bindings:
     PDT_PRG_003M: {backend: extracted/PDT/backend/pdt_pgr_003m,
                    frontend: extracted/PDT/frontend/feature-prg-003}
   ```
2. **`id-code`** вҖ” screen-id matches a module dir (exact, or transposition-tolerant so
   `PDT_PRG_003M вҶ’ pdt_pgr_003m`). Digits are part of the multiset match, so **003 no longer
   collides with 004** (the original over-binding bug).
3. **`sp-fallback`** вҖ” specificity-weighted (inverse-document-frequency) stored-proc matching,
   used only when the id maps to no module. A unique SP is decisive; a shared generic SP counts
   for little. This is the only path that can still over-bind, so when it is used the audit
   sets `resolution.uncertain: true`, emits a `ScreenResolutionLowConfidence` finding telling
   the reviewer to pin an explicit `screen_binding`, and downgrades every other finding to
   `confidence: low` + `resolution_uncertain: true`.

So heuristic findings are still produced but are **honestly flagged as untrustworthy** until an
explicit binding is added вҖ” no longer silently presented as reliable.

## Labels dimension (opt-in vision) вҖ” `--labels`

The нҷ”л©ҙ Layout screenshot in the spec carries Korean field labels (мһ…кі л…„мӣ”, н’ҲлӘ©мҪ”л“ң вҖҰ) that
exist nowhere in cell text. `erpqa screen-audit вҖҰ --labels` reads them **for free, on-device**
via the macOS **Vision** framework (a tiny Swift helper compiled once; no API, no cost, good
Korean accuracy). Each label is checked against the *whole* frontend slice (labels live in
header-config / shared models, not just the screen cluster); a label shown in the spec whose
Korean string appears nowhere in the frontend is flagged `LabelMissingInFrontend`.

Properties:
- **Opt-in & non-deterministic** вҖ” off by default; the core audit stays deterministic/offline.
  On non-macOS hosts (or without `swiftc`) OCR degrades to empty and the dimension reports
  `ocr_available: false` instead of failing.
- **OCR is noisy** вҖ” every label finding is `MINOR` / `confidence: low` / `needs_human_confirmation`.
  Data the OCR mistakes for labels (codes, amounts, `(мЈј)вҖҰ` company names, parenthesised pivot
  values) is filtered by `is_label`, but human review of the screenshot is still required.
- **EMF/WMF** embedded images (vector metafiles, no offline converter) are manifested as skipped,
  not OCR'd. The main layout screenshots in these specs are PNG and are read directly.
- Validated on PDT-OSC-001M: 26 labels read, 20 matched, 6 human-review candidates.

## Known limitations (do NOT trust yet)
- **Spec coverage gaps вҖ” now surfaced, not silent.** Some л“ұлЎқ/мһ…л Ҙ/мһ‘м—…ліҙкі  specs carry no
  save-SP (`_IU`) example, so the save dimension cannot be evaluated. The audit now emits a
  `SaveSpecExampleMissing` finding when a screen looks like a save screen (its name says
  л“ұлЎқ/мһ…л Ҙ/мһ‘м—…ліҙкі  вҖ” matched after NFC-normalizing the macOS-NFD filename вҖ” or the backend
  implements `IU`, or the frontend has a create/update mutation) yet no save example exists,
  instead of silently reporting CLEAN. Numbered SP variants (`_S2`/`_S4`/`_IU2`) now map to
  their base role (was lost as OTHER). Param spelling (`IYM`/`iYM`) is handled.
- **Search-filter / grid info** depend on the frontend slice including shared `src/models/`.

## Input requirement (copy-step rule) вҖ” now enforced

When assembling a module-scoped extracted folder, the frontend slice **must include the shared
`src/models/`** (and `src/adapters/`), not just `feature-*/`. Query screens import their zod
models from `@/models/...` and their api clients from `@/adapters/...`; without these the audit
cannot read frontend fields.

The audit now **enforces** this: while following a seed file's imports it records every
`@/models` / `@/adapters` import that resolves to nothing in the slice (file *or* directory/index
вҖ” path-fragment matched, so directory-organized modules like `models/sales-summary/` are not
false-flagged) and emits a `FrontendSliceIncomplete` finding naming the exact missing specifiers.
This is the precise, actionable cause behind most `FrontendModelUnresolved` cases. On the PDT
clone it correctly caught 5 screens whose `@/adapters/api/...` submodules
(`production-plan-management`, `view-config`) were left out of the copy вҖ” a partial copy-step
violation (shared `models/` was copied, some shared `adapters/` were not).
