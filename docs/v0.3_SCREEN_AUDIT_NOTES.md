# v0.3 (WIP) — Screen-Scoped Audit

Status: **prototype, validated on the real pipe-manufacturing ERP (module PDT)**.
This is additive to v0.2 and does not change v0.1/v0.2 behavior. It exists because the
v0.2 module-level extractor could not handle real screen-spec xlsx (formatted documents,
not clean tables) and modelled one screen per module — see the findings below.

## Command

```
erpqa screen-audit <project_path> --module <MOD> --screen <SCREEN_ID>
```

Resolves three sources for one screen and compares them:
- **spec** — the screen's xlsx 화면IO sheet. The reliable field source in this ERP is the
  embedded stored-procedure examples (`exec ..._IU @param=` for save, `..._S` for query).
- **backend** — the screen's module; pydantic `serialization_alias`/`validation_alias` and the
  service param dicts bridge SP params (`iInQty`) ↔ clean keys (`in_qty`) ↔ DB cols (`INQTY`).
- **frontend** — the feature file cluster (model + hooks), reached by following imports.

Output: `qa-context/modules/<MOD>/screens/<SCREEN_ID>/screen_audit{.yaml,_report.md}`.

## What is trustworthy (verified true positives)

- **Save-field comparison** (등록 screens): spec save-SP params vs frontend model fields.
  Verified on `PDT-OSC-001M` (found `pg_no`, `add_emp_no` missing in FE), `PDT-OSC-002M`,
  `PDT-OSC-004M`. Server-populated fields (audit/document-no) are flagged MINOR +
  `needs_human_confirmation: true`.
- **CLEAN detection**: when filters + columns fully match, the screen is reported consistent
  (e.g. `PDT-OSC-003M`, several `MG` screens).
- **Honesty guards**:
  - If the frontend model is not resolved (e.g. shared `src/models/` missing from the slice),
    emit ONE `FrontendModelUnresolved` notice instead of dozens of false "missing".
  - `BackendColumnNotInFrontend` is **info only**, never a finding — backend is *evidence*,
    not source of truth, and grids legitimately show a subset of returned columns.
  - Dynamic pivot columns (`specNN`/`dayNN`) are excluded from per-column comparison.

## Known limitations (do NOT trust yet)

- **Resolution precision (the main blocker).** Screens that share generic stored procedures
  (e.g. the `PRG`/진행현황 group) can over-bind to the same backend module via SP-name
  matching (`PDT_PRG_003M` and `PDT_PRG_004M` both resolved to `pdt_pgr_001m`). Their findings
  (incl. `SaveCapabilityMissing`) are therefore **not reliable**. Fix direction: bind each
  screen to exactly one backend/frontend via its **unique/primary SP** or a maintained
  `screen_id → backend_dir → fe_feature` mapping.
- **Labels / layout not read.** The 화면 Layout and visible Korean labels live in embedded
  screenshots (WMF). No OCR/vision is done — a future dimension.
- **Spec coverage gaps.** Some 등록 specs have no save-SP example; some use inconsistent param
  spelling (`IYM`/`iYM`, handled) or abbreviations.
- **Search-filter / grid info** depend on the frontend slice including shared `src/models/`.

## Input requirement (copy-step rule)

When assembling a module-scoped extracted folder, the frontend slice **must include the shared
`src/models/`** (and `src/adapters/`), not just `feature-*/`. Query screens import their zod
models from `@/models/...`; without these the audit cannot read frontend fields and will report
`FrontendModelUnresolved`. Update the Codex copy guideline accordingly.
