# ERP QA Kit v0.2 — Dry-Run Guide (PM/QA Edition)

> **Planning document.** Describes how to run v0.2 once it is implemented. No code here; nothing in this guide modifies your ERP.

---

## 1. Purpose & who this is for

This guide is for a **non-developer PM or QA lead** who needs to check whether a built ERP **screen** matches its **screen specification** — starting with the **ORD (Order)** module of the pilot.

You do **not** need to read code, write code, or touch a database. You assemble a small input folder, run a few commands, and read a plain-language mismatch report. A developer (or an AI coding agent like Codex/Claude) handles any fixes afterward.

**Core principle of the kit:** *LLMs propose; deterministic extractors/checks verify; humans (you) approve.* The kit suggests; it never silently changes anything; you make the final call.

This is a **safe dry run**:
- It is **read-only with respect to the ERP.** It reads the ORD source files you hand it and writes its own working notes elsewhere.
- It changes **nothing** in the ERP repositories, the running application, or any database.
- Worst case, you delete the kit's output folder and nothing else is affected.

---

## 2. Safety guarantees (read this first)

- [ ] **No database access.** The kit never connects to, queries, or executes anything against any DB. DB/business-logic checks are **deferred** in v0.2 (see §13).
- [ ] **No changes to target ERP code.** The kit never edits, deletes, formats, or commits anything in the ERP repos. It only **reads** the ORD files you give it.
- [ ] **Writes only under `qa-context/`.** Every artifact the kit produces (contracts, reports, handoffs, memory) lands inside the kit's own `qa-context/` working area. Nothing is written into the ERP source tree.
- [ ] **Module-scope only.** You run against **one module (ORD)** at a time, using a **small extracted folder** — never the whole repo, never all repos.
- [ ] **Human gates everywhere.** Anything uncertain (image-derived text, low-confidence items, frontend-vs-policy conflicts) is flagged `needs_human_confirmation:true` and waits for **you**.

If any of these guarantees appears violated during a run, **stop** and treat it as a bug.

---

## 3. Pre-flight checklist

**Do NOT clone all repos.** The pilot ERP has **7 modules (only 1 verified)** across **3 repositories**. The full repos are too large to clone or analyze at once. Instead, assemble a **small MODULE-SCOPED EXTRACTED FOLDER** containing only ORD material.

- [ ] You have the **ORD screen-spec file(s)** (the source of truth for visible text/labels/columns/search filters/form fields).
- [ ] You have the **ORD frontend files/folder** (the implementation under test).
- [ ] You have the **ORD backend files/folder** (supporting evidence — *not* fully trusted).
- [ ] You have any **ORD stored-procedure `.sql` files** (may reflect *intended* logic, but **unverified**).
- [ ] You have **nothing else** — no unrelated modules, no full repo trees, no node_modules/build output.
- [ ] Python with **openpyxl** is available if your spec is xlsx (see §14).

**Screen-spec formats supported in v0.2:**

| Format | Supported in v0.2 | Notes |
|---|---|---|
| xlsx (text) | Yes (required adapter) | Plain cells extracted as high/medium confidence |
| xlsx with screenshots/images | Yes (required adapter) | Produces an **image MANIFEST only — no OCR**; flagged `needs_human_confirmation:true` |
| csv | Yes (required adapter) | |
| markdown | Yes (required adapter) | |
| json / yaml | Yes (required adapter) | |
| pdf / docx / pptx / figma | **No** (future/optional adapters) | Required flow still works without them |

> **Reminder:** the **screen spec + backend evidence** are the source of truth for visible text/labels/columns/filters/form fields. The **frontend is the thing being tested**, not the answer key.

## 4. Preparing the extracted folder

Build one small folder that holds *only* ORD inputs. Example layout:

```text
ord-extract/
├── spec/
│   └── ORD_screen_spec.xlsx          # or .csv / .md / .json / .yaml
├── frontend/
│   └── ord/                          # ORD frontend files ONLY (under test)
├── backend/
│   └── ord/                          # ORD backend files ONLY (evidence)
└── procedure/
    └── ord/
        └── *.sql                     # ORD stored procedures (unverified)
```

Because the three real repositories live in different places, you tell the kit **where each piece comes from** via the repository/path mapping in `project_policy.yaml` (created in §5). Conceptually:

```yaml
# project_policy.yaml (excerpt — repository / path mapping)
repositories:
  frontend_repo:  { path: "ord-extract/frontend" }
  backend_repo:   { path: "ord-extract/backend" }
  procedure_repo: { path: "ord-extract/procedure" }

modules:
  ORD:
    spec:      "ord-extract/spec/ORD_screen_spec.xlsx"
    frontend:  "frontend_repo/ord"
    backend:   "backend_repo/ord"
    procedure: "procedure_repo/ord"
```

This mapping is how multi-repo paths are resolved to your one extracted folder. You never point the kit at a full repo root.

## 5. Step 1 — `policy-init` (project level)

Creates the project-level policy and memory files. Run once per project.

```bash
qa-kit policy-init <project_path>
```

Produces under `qa-context/`:
- `project_policy.yaml` — machine-readable rules
- `project_memory.md` — PM-readable running notes
- `project_assumptions.yaml` — tracked assumptions awaiting confirmation

**Review & confirm the pilot policy:**

- [ ] `strict_screen_contract_first: true` — screen spec leads; frontend is validated against it.
- [ ] `frontend_override_allowed: false` — **FRONTEND OVERRIDE FORBIDDEN.** The frontend never wins a disagreement with the spec.
- [ ] `validation_order:` — frontend screen correctness is **first**; procedure/business-logic come later.
- [ ] `defer:` list includes **procedure** and **business-logic** validation until frontend mismatches are resolved.
- [ ] `project_memory.md` accurately reflects the pilot reality (7 modules / 1 verified, 3 repos, backend not fully trusted, procedures unverified, frontend = implementation under test).

> If `project_policy.yaml` is **missing**, a run **halts** (or creates a *draft* that **requires your confirmation** before proceeding). See §14.

---

## 6. Step 2 — `module-init --module ORD`

Scaffolds the ORD module workspace.

```bash
qa-kit module-init <project_path> --module ORD
```

Produces (under `qa-context/`):
- `module_policy.yaml` — ORD-specific policy (inherits project policy)
- `module_memory.md` — ORD-specific notes

- [ ] Confirm `module_policy.yaml` inherits `frontend_override_allowed:false` and the deferral list.
- [ ] Confirm `module_memory.md` names ORD as the first module under test and records the spec format you supplied.

---

## 7. Step 3 — run extraction

Extraction normalizes each input into a contract file, with every item carrying **source / confidence / needs_human_confirmation**:

- `screen_contract.yaml` — from the screen spec (source of truth)
- `frontend_contract.yaml` — from the ORD frontend (under test)
- `backend_contract.yaml` — from the ORD backend (evidence)
- `procedure_contract.yaml` — from ORD stored procedures (unverified)

**Option A — granular commands** (run individually, inspect between):

```bash
qa-kit extract-spec      <project_path> --module ORD
qa-kit extract-frontend  <project_path> --module ORD
qa-kit extract-backend   <project_path> --module ORD
qa-kit extract-procedure <project_path> --module ORD   # deferred analysis; extraction only
```

**Option B — let the orchestrator do it** (see §10).

> **xlsx with images:** the image adapter produces an **image MANIFEST, not OCR**. Any text you believe is *in* a screenshot is **not** auto-read; the manifest items are flagged `needs_human_confirmation:true` and must be reviewed by you (§11–12) before they are trusted.

---

## 8. Step 4 — `compare-contract` & `module-report`

```bash
qa-kit compare-contract <project_path> --module ORD
qa-kit module-report    <project_path> --module ORD
```

Reads the contracts and emits a plain-language **mismatch report**. Common mismatch types:

| Mismatch type | Plain meaning |
|---|---|
| **Missing in frontend** | Spec requires a label/column/filter/field that the frontend does **not** show → frontend must add it. |
| **Extra in frontend** | Frontend shows something the spec does **not** define → flagged. Do **not** silently accept it (§13). |
| **Text/label mismatch** | Same field, different wording → frontend wording must match the spec. |
| **Order/structure mismatch** | Columns/fields present but in the wrong order/grouping vs spec. |
| **Low-confidence / image-derived** | Item came from an image manifest or low-confidence source → confirm before acting. |

**What "frontend override forbidden" means for your decisions:** when frontend and spec disagree, the **spec wins**. You never resolve a mismatch by declaring the frontend correct and editing the spec to match it. The fix direction is always *frontend → conform to spec* (with backend as supporting evidence).

The report **cites that policy + memory were read** at the start of the run — confirm this line is present.

---

## 9. Step 5 — `module-handoff`

```bash
qa-kit module-handoff <project_path> --module ORD
```

Produces handoff artifacts under `qa-context/`:
- `frontend_fix_handoff.md` — human-readable summary of confirmed frontend mismatches
- `codex_frontend_fix_prompt.md` — a ready-to-paste prompt for an AI coding agent

Hand `codex_frontend_fix_prompt.md` (or the handoff doc) to **Codex/Claude** to fix the **TARGET FRONTEND repo** — **not the kit**. The kit never edits target code itself; it only describes what needs to change so a developer or coding agent can apply it in the real frontend repository.

- [ ] Only **confirmed** mismatches (you have approved them) should drive fixes.
- [ ] Fixes go to the **frontend** (override forbidden — never "fix" the spec to match the build).

---

## 10. One-shot alternative — `module-audit --module ORD`

Runs everything above in the policy's `validation_order`, and **skips the deferred stages** (procedure / business-logic) until the frontend is verified.

```bash
qa-kit module-audit <project_path> --module ORD
```

This is equivalent to: read policy+memory → `extract-spec` → `extract-frontend` → `extract-backend` → `compare-contract` → `module-report` → `module-handoff`, with procedure/business-logic validation held back. Use the granular commands (§7) when you want to inspect each step; use `module-audit` for a clean end-to-end pass.

---

## 11. Interpreting confidence & `needs_human_confirmation`

Every contract item carries a confidence level. Use this PM decision table:

| Confidence | Typical source | Your action |
|---|---|---|
| **High** | Plain xlsx/csv/md/json text; clear static frontend code | Trust as a baseline; still sanity-check flagged conflicts. |
| **Medium** | Ambiguous cells, inferred structure | Skim and confirm before it drives a fix. |
| **Low** | Image-derived / OCR-style / weak signals | **Always human-confirm before acting.** |

**Rule:** any item with `needs_human_confirmation:true` — which **always** includes image-derived / xlsx-image-manifest text and any low-confidence item — must be **confirmed by you** before it is trusted or used to justify a frontend fix.

---

## 12. Human review gates

Before you approve a handoff, you (the PM/QA lead) **must** confirm:

- [ ] **Source-of-truth correctness** — the screen-spec contract really reflects the intended screen.
- [ ] **Low-confidence extractions** — reviewed and accepted or corrected.
- [ ] **Image-derived text** — every image-manifest item checked against the actual screenshot (no OCR was done for you).
- [ ] **Frontend-vs-policy conflicts** — any place the frontend disagrees with the spec is resolved **in the spec's favor**.

---

## 13. What NOT to do

- [ ] **Don't trust the frontend as truth.** It is the implementation under test, not the answer key.
- [ ] **Don't accept extra frontend columns/fields/labels** that aren't in the spec just because they exist.
- [ ] **Don't run procedure / business-logic validation before frontend mismatches are resolved.** These are deferred by policy.
- [ ] **Don't clone whole repos** or pull in other modules — ORD module-scope extracted folder only.
- [ ] **Don't expect DB checks** — there are none in v0.2.
- [ ] **Don't edit the spec to match the build** — frontend override is forbidden.

---

## 14. Troubleshooting

| Symptom | What it means / fix |
|---|---|
| Run **halts: missing `project_policy.yaml`** | Run `policy-init` first, or confirm the auto-generated draft policy before proceeding (§5). |
| `openpyxl` not installed | Install it for xlsx specs: `pip install openpyxl` — then re-run `extract-spec`. |
| Optional adapter absent (pdf/docx/pptx/figma) | Expected in v0.2. Convert/export your spec to a **required** format (xlsx/csv/md/json/yaml). The required flow still works. |
| Report doesn't cite policy/memory | The run didn't read policy+memory — treat as a bug and re-run after confirming `qa-context/` is intact. |
| Frontend "extra" items everywhere | Likely you pointed at too broad a folder — re-check the ORD module-scoped extract (§3–4). |

---

## 15. After the dry run

Close the loop so the next run is smarter:

- [ ] Record confirmed findings, accepted low-confidence items, and any image-text decisions into **`module_memory.md`** (ORD).
- [ ] Promote durable, project-wide decisions and confirmed/retired assumptions into **`project_assumptions.yaml`** and **`project_memory.md`**.
- [ ] Note which mismatches were handed off and their fix status, so the next ORD run starts from current reality.

Every future run reads policy + memory **first**, so these notes directly steer subsequent audits.
