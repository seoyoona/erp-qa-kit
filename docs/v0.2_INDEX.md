## Non-Negotiable Rules for v0.2 (restated)

1. **Dependencies:** `openpyxl` is the **only** new required runtime dependency for v0.2. All
   other required adapters use the Python standard library plus the existing `PyYAML`.
2. **Optional extras:** `pandas` and **MarkItDown** are **optional extras only**, and only if
   a **graceful fallback is implemented** (the required flow must still work when they are not
   installed). They must never become required.
3. **Future-only:** **Docling** and **Unstructured** are **future-only** and **must not be
   implemented in v0.2** ‚Äî document only.
4. **No placeholder extractors:** every extractor marked *required* in
   `v0.2_EXTRACTOR_STRATEGY.md` must be **real, wired to its CLI command, covered by
   tests/fixtures, and documented**. Placeholder-only folders are a failure.
5. **Read policy & memory first:** every `module-audit` command (and every granular module
   command and skill) **must read `project_policy.yaml`, `project_memory.md`,
   `module_policy.yaml`, and `module_memory.md` before doing any work**. If
   `project_policy.yaml` is missing, the run halts or creates a draft requiring human
   confirmation; reports/handoffs must cite that memory was read.
6. **Frontend is implementation-under-test, not source of truth.** The source of truth for
   visible text, labels, columns, search filters, and form fields is the **screen spec +
   backend evidence**.
7. **Frontend override is forbidden** when policy says `frontend_override_allowed: false`. A
   frontend-vs-policy conflict is **reported, never silently resolved**.
8. **Uncertain mappings:** any uncertain interpretation or mapping must be marked
   **`needs_human_confirmation: true`**. OCR/image-derived text is capped at low/medium
   confidence and always `needs_human_confirmation: true` (never high).
9. **No v0.1 regression:** v0.2 is purely additive. All v0.1 commands, file formats, and the
   existing 19 tests must keep working unchanged. No DB execution; no modification of target
   ERP code; writes only under `qa-context/`.

## Terminology Clarification (do NOT rename)

The two `frontend_override` terms are **intentionally distinct** and must be kept as-is:

- **`frontend_override_allowed`** ‚Äî the **policy field** in `project_policy.yaml` /
  `module_policy.yaml` (per source-of-truth aspect). Boolean; `false` means the frontend may
  not override that aspect.
- **`frontend_override_forbidden`** ‚Äî the **derived report field** emitted per mismatch
  finding. It is the negation of the effective policy value
  (`frontend_override_forbidden == not frontend_override_allowed`), surfaced so a PM reading a
  report sees the decision directly.

These are not duplicates and **must not be renamed or merged**. Keep both spellings exactly.
