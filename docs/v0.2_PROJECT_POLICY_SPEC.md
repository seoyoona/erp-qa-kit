# ERP QA Kit v0.2 — Project & Module Policy Specification

**Status:** Planning (PLANNING ONLY — no code in this doc; no v0.1 files modified).
**Audience:** Codex (implementation), PM (review/approval).
**Authoritative for:** `qa-context/project_policy.yaml` and `qa-context/modules/{MODULE}/module_policy.yaml`.
**Sibling docs (do not duplicate here):** `v0.2_SPEC.md`, `v0.2_PROJECT_MEMORY_SPEC.md`, `v0.2_EXTRACTOR_STRATEGY.md`, `v0.2_ARCHITECTURE.md`, `v0.2_CONSTRAINTS.md`, `v0.2_ACCEPTANCE_CRITERIA.md`.

This document defines the **schema, semantics, validation, and behavior contract** for the two policy files. The *commands* that read them, the *contract* shapes they produce, and the *extractor implementations* are owned by the sibling docs above and are referenced, not restated.

---

## 0. Core principle (binding)

> **Project policy is machine-readable and drives every command.**
> LLMs **propose**; deterministic extractors/checks **verify**; humans **approve**.

The toolkit **never** modifies target ERP code, **never** executes DB queries, and writes **only** under `qa-context/`. The frontend is the **implementation under test, not the source of truth**. Conflicts between the frontend and policy/source-of-truth are **reported, never silently resolved**. When policy forbids frontend override, that prohibition is **strictly enforced**. Any uncertain interpretation sets `needs_human_confirmation: true`.

These rules are not advisory text inside reports — they are encoded as **fields** in `project_policy.yaml` that every command loads and obeys *before* doing work.

---

## 1. Purpose & scope

`project_policy.yaml` is the single machine-readable description of **how this ERP project is QA'd**. It is:

1. **Loaded before every command** runs (see §6 load order). No command does QA work before reading it.
2. **The driver of command behavior** — it determines source-of-truth precedence, whether frontend may override, the order validation steps run in, which validations are deferred, what assumptions are forbidden, which file formats and extractors are allowed, and how OCR/image-derived confidence is capped.
3. **Per-project, reusable across many ERP projects** — the pilot is strict; other projects may be laxer (see §8b). The schema generalizes.

`module_policy.yaml` **refines and overrides** the project policy for a single module, but **may never relax** a project-level prohibition (see §5, §7).

Out of scope for this doc (owned elsewhere):
- Command synopses, exit codes, idempotency — `v0.2_SPEC.md`.
- Contract file shapes (`screen_contract.yaml`, `frontend_contract.yaml`, …) — `v0.2_SPEC.md`.
- Memory files (`project_memory.md`, `module_memory.md`, `project_assumptions.yaml`) — `v0.2_PROJECT_MEMORY_SPEC.md`.
- Concrete extractor/adapter implementation, deps, lazy-loading — `v0.2_EXTRACTOR_STRATEGY.md`.
- Component/registry wiring — `v0.2_ARCHITECTURE.md`.

## 2. File locations & precedence

| File | Path | Scope | Written by |
|---|---|---|---|
| Project policy | `qa-context/project_policy.yaml` | Whole ERP project | `erpqa policy-init <project_path>` |
| Module policy | `qa-context/modules/{MODULE}/module_policy.yaml` | One module | `erpqa module-init <project_path> --module {MODULE}` |

Both live **inside `qa-context/`** and are therefore covered by the v0.1 write-containment guard `assert_within_qa_context`. The toolkit never writes a policy file outside `qa-context/`.

**Precedence (lowest → highest binding):**

```
project_policy.yaml   →   module_policy.yaml   →   (effective policy for this module)
```

- A module policy key that is **present** overrides the same project key **for that module only**.
- A module policy key that is **absent** inherits the project value.
- Override is allowed only in the **stricter or additive** direction (see §5/§7). A module may *tighten* but never *loosen* a project prohibition.
- Lists like `forbidden_assumptions` and `validation_order.defer_until_frontend_verified` are **merged additively** (module entries are appended/unioned, never used to delete project entries).

## 3. The pilot example (verbatim)

This is the **strict pilot** policy (`strict_screen_contract_first`), reproduced exactly. It corresponds to the pilot reality: 7 modules (1 verified), 3 repos including ORD, frontend override strictly forbidden, frontend-screen-correctness first, procedure/business-logic validation deferred.

```yaml
qa_policy:
  name: strict_screen_contract_first
  review_strategy: module_by_module
  source_of_truth:
    visible_text:
      primary: [screen_spec, backend]
      secondary: [frontend]
      frontend_override_allowed: false
    grid_columns:
      primary: [screen_spec, backend]
      secondary: [frontend]
      frontend_override_allowed: false
    form_fields:
      primary: [screen_spec, backend]
      secondary: [frontend]
      frontend_override_allowed: false
    search_filters:
      primary: [screen_spec, backend]
      secondary: [frontend]
      frontend_override_allowed: false
    procedure_mapping:
      primary: [backend, procedure]
      secondary: [frontend, screen_spec]
      frontend_override_allowed: false
  validation_order:
    - read_project_policy
    - read_project_memory
    - read_module_policy
    - read_module_memory
    - screen_spec_extraction
    - frontend_contract_extraction
    - screen_vs_frontend_comparison
    - backend_mapping_check
    - procedure_mapping_check
    - data_flow_validation
    - procedure_business_logic_validation
  defer_until_frontend_verified:
    - procedure_business_logic_validation
    - cross_module_data_flow_validation
  forbidden_assumptions:
    - Do not treat frontend implementation as correct if it conflicts with screen specs.
    - Do not silently accept extra frontend columns.
    - Do not override screen spec labels with frontend labels.
    - Do not assume backend/procedure is correct without explicit evidence.
```

> The pilot above is the *minimum* a real policy carries for QA semantics. §4 defines **additional required blocks** (`allowed_file_formats`, `extraction_confidence_rules`, `human_confirmation_requirements`, `module_scope`, `repository_path_mapping`, `extractors`, `frontend_override`) that `policy-init` scaffolds alongside it. §8a shows the pilot with those blocks filled in.

### 3.1 Explanation of each part of the pilot

| Part | What it does |
|---|---|
| `name` | Human-readable policy identity used in reports/handoffs. |
| `review_strategy: module_by_module` | Drives the audit loop: one module at a time, no full-repo sweep (matches v0.2 anti-overengineering rule). |
| `source_of_truth.<aspect>.primary` | Ordered list of trusted sources for that aspect. First listed is most authoritative. |
| `source_of_truth.<aspect>.secondary` | Corroborating-only sources. They can *agree/conflict* but cannot *decide*. `frontend` sits here for every aspect. |
| `source_of_truth.<aspect>.frontend_override_allowed: false` | Frontend may never overrule the primary sources for that aspect — strictly enforced by the comparison engine. |
| `procedure_mapping` precedence `[backend, procedure]` | For procedure mapping, backend + client stored procedure lead; frontend and screen_spec are only secondary (procedures may reflect intended logic but are unverified). |
| `validation_order` | Exact step sequence the module-audit pipeline executes; policy + memory reads come first. |
| `defer_until_frontend_verified` | Steps that must **not** run as authoritative until frontend mismatches are resolved (procedure business logic, cross-module data flow). |
| `forbidden_assumptions` | Hard guardrails the LLM proposer must not violate; checked and surfaced in reports. |

## 4. `project_policy.yaml` — field-by-field reference

All fields live under the top-level `qa_policy:` mapping. `<aspect>` ranges over the fixed set: `visible_text`, `grid_columns`, `form_fields`, `search_filters`, `procedure_mapping`. Source tokens range over the fixed set: `screen_spec`, `backend`, `procedure`, `frontend`.

### 4.1 Identity & strategy

| Field | Type | Allowed values | Required | Meaning | Default (scaffold) |
|---|---|---|---|---|---|
| `name` | string | free text, non-empty | yes | Policy identity, shown in reports. | `strict_screen_contract_first` |
| `review_strategy` | enum string | `module_by_module` (v0.2 only supported value; `full_project` reserved/future) | yes | Audit granularity; module-by-module forbids full-repo sweeps. | `module_by_module` |

### 4.2 `source_of_truth.<aspect>`

| Field | Type | Allowed values | Required | Meaning | Default |
|---|---|---|---|---|---|
| `source_of_truth` | map | keys = the 5 aspects | yes | Per-aspect precedence + override flag. | (all 5 present) |
| `source_of_truth.<aspect>.primary` | list<string> | subset of source tokens, ordered, non-empty | yes | Authoritative sources; index 0 = strongest. | text/grid/form/filters: `[screen_spec, backend]`; procedure_mapping: `[backend, procedure]` |
| `source_of_truth.<aspect>.secondary` | list<string> | subset of source tokens | yes | Corroborating-only sources (agree/conflict, never decide). | text/grid/form/filters: `[frontend]`; procedure_mapping: `[frontend, screen_spec]` |
| `source_of_truth.<aspect>.frontend_override_allowed` | boolean | `true` / `false` | yes | Per-aspect: may frontend overrule primary for this aspect? | `false` |

A source token must not appear in both `primary` and `secondary` of the same aspect.

### 4.3 Ordering & deferral

| Field | Type | Allowed values | Required | Meaning | Default |
|---|---|---|---|---|---|
| `validation_order` | list<string> | ordered steps from the **step vocabulary** below | yes | Exact pipeline step order; consumed by module-audit. | the pilot list (§3) |
| `defer_until_frontend_verified` | list<string> | subset of `validation_order` step tokens | yes (may be empty `[]`) | Steps that must not run authoritatively until frontend is verified. | `[procedure_business_logic_validation, cross_module_data_flow_validation]` |

**Step vocabulary (closed set for v0.2):** `read_project_policy`, `read_project_memory`, `read_module_policy`, `read_module_memory`, `screen_spec_extraction`, `frontend_contract_extraction`, `screen_vs_frontend_comparison`, `backend_mapping_check`, `procedure_mapping_check`, `data_flow_validation`, `procedure_business_logic_validation`, `cross_module_data_flow_validation`. The first four (policy/memory reads) MUST appear, in that relative order, at the start of `validation_order`.

### 4.4 Assumptions & formats

| Field | Type | Allowed values | Required | Meaning | Default |
|---|---|---|---|---|---|
| `forbidden_assumptions` | list<string> | free-text guardrails | yes (may be empty `[]`) | Hard rules the LLM proposer must not violate; surfaced in reports. | the 4 pilot lines (§3) |
| `allowed_file_formats` | list<string> | file extensions w/o dot, lowercase | yes | Whitelist of spec/source file formats extractors may ingest. | `[xlsx, csv, md, json, yaml, yml, sql]` |

### 4.5 `extraction_confidence_rules`

Governs how confidence (`high|medium|low`) and `needs_human_confirmation` are assigned. (Per-item propagation in comparisons is owned by `v0.2_SPEC.md` §5; this block sets the **caps and defaults** policy enforces.)

| Field | Type | Allowed values | Required | Meaning | Default |
|---|---|---|---|---|---|
| `extraction_confidence_rules` | map | — | yes | Confidence policy. | (below) |
| `extraction_confidence_rules.default_needs_human_confirmation` | boolean | `true`/`false` | no | Default for newly extracted items lacking a human decision. | `true` |
| `extraction_confidence_rules.ocr_image` | map | — | yes | Rules for OCR / image-derived text. | (below) |
| `…ocr_image.usable` | boolean | `true`/`false` | yes | May OCR/image-derived text be used at all? | `true` |
| `…ocr_image.max_confidence` | enum string | `low` / `medium` (never `high`) | yes | Hard cap on confidence for image-derived text; `high` is rejected by validation. | `medium` |
| `…ocr_image.needs_human_confirmation` | boolean | must be `true` | yes | Image-derived items always require human confirmation. | `true` |

> Per the shared brief: OCR/image-derived text is **never** `high` and **always** `needs_human_confirmation: true`. There is no `xlsx_image_manifest` OCR in v0.2 (image manifest only, no OCR) — see `v0.2_EXTRACTOR_STRATEGY.md`; the cap exists so any future OCR/`docling`/`unstructured` path is governed from day one.

### 4.6 `human_confirmation_requirements`

| Field | Type | Allowed values | Required | Meaning | Default |
|---|---|---|---|---|---|
| `human_confirmation_requirements` | map | — | yes | When human confirmation is mandatory. | (below) |
| `…require_for_low_confidence` | boolean | `true`/`false` | no | Any `low`-confidence item must be human-confirmed. | `true` |
| `…require_for_missing_policy_draft` | boolean | must be `true` | no | A DRAFT policy created on missing-policy halt requires human confirmation before any QA work. | `true` |
| `…require_for_frontend_conflict` | boolean | `true`/`false` | no | A reported frontend↔primary conflict is held for human decision (never auto-resolved). | `true` |
| `…require_for_deferred_areas` | boolean | `true`/`false` | no | Deferred validations, when finally run, require human confirmation. | `true` |

### 4.7 `module_scope`

Defines how a module name maps to extracted-folder paths. Module-scoped extracted folders are **preferred over full clones** (pilot reality).

| Field | Type | Allowed values | Required | Meaning | Default |
|---|---|---|---|---|---|
| `module_scope` | map | — | yes | Module → path mapping policy. | (below) |
| `…strategy` | enum string | `extracted_folder` (preferred) / `full_repo` | yes | How module sources are located; `extracted_folder` = module-scoped folder, no full clone. | `extracted_folder` |
| `…module_path_template` | string | template w/ `{repo_root}`, `{module}` placeholders | yes | How to compute a module's folder per repo. | `{repo_root}/modules/{module}` |
| `…spec_glob` | list<string> | glob patterns | no | Where screen-spec source files live for a module. | `["specs/**/*.xlsx", "specs/**/*.md"]` |
| `…require_full_clone` | boolean | should be `false` | no | If `true`, full clone is required — discouraged. | `false` |

### 4.8 `repository_path_mapping` (multi-repo)

The pilot ERP project spans **3 repositories including ORD**. This block maps each repo name to its on-disk path / extracted folder.

| Field | Type | Allowed values | Required | Meaning | Default |
|---|---|---|---|---|---|
| `repository_path_mapping` | list<map> | one entry per repo | yes (≥1 entry) | Repo → path/extracted-folder, with role. | (3-repo scaffold below) |
| `repository_path_mapping[].repo` | string | repo name (e.g. `ORD`) | yes | Logical repo identifier. | — |
| `repository_path_mapping[].role` | enum string | `frontend` / `backend` / `procedure` / `mixed` | yes | What kind of evidence this repo provides. | — |
| `repository_path_mapping[].path` | string | path or `extracted-folder://…` | yes | On-disk location or module-scoped extracted folder. | — |
| `repository_path_mapping[].extracted_folder` | boolean | `true`/`false` | no | `true` = module-scoped extract (preferred), no full clone. | `true` |
| `repository_path_mapping[].trusted` | boolean | `true`/`false` | no | Backend repos: `false` (not fully trusted); procedure repos: `false` (intended-but-unverified). | backend/procedure: `false` |

Three-repo scaffold (placeholder names; ORD is real):

```yaml
  repository_path_mapping:
    - repo: ORD            # order module repo (real pilot repo)
      role: mixed
      path: extracted-folder://ORD
      extracted_folder: true
      trusted: false
    - repo: REPO_FRONTEND  # frontend implementation under test
      role: frontend
      path: extracted-folder://REPO_FRONTEND
      extracted_folder: true
      trusted: false       # frontend is implementation-under-test, never SoT
    - repo: REPO_BACKEND   # backend evidence (corroborating, not fully trusted)
      role: backend
      path: extracted-folder://REPO_BACKEND
      extracted_folder: true
      trusted: false
```

### 4.9 `extractors` config block

Gates the adapter registry (see `v0.2_ARCHITECTURE.md` / `v0.2_EXTRACTOR_STRATEGY.md`). Classifies extractors as **required / optional / future** and sets per-extractor enable + allowed extensions.

| Field | Type | Allowed values | Required | Meaning | Default |
|---|---|---|---|---|---|
| `extractors` | map | keys `required` / `optional` / `future` | yes | Extractor tiers. | (below) |
| `extractors.required` | list<map> | required extractors | yes | Must be real, wired, and tested; gating registry refuses to run if a required one is missing. | the 9 required (below) |
| `extractors.optional` | list<map> | optional extractors | no | Enabled only if dep present + `enabled: true`. | `pandas`, `markitdown` |
| `extractors.future` | list<map> | reserved extractors | no | Declared, never auto-enabled in v0.2. | `docling`, `unstructured` |
| `extractors.*[].name` | string | extractor id | yes | e.g. `xlsx_text`. | — |
| `extractors.*[].enabled` | boolean | `true`/`false` | yes | Whether the adapter is active. | required: `true`; future: `false` |
| `extractors.*[].extensions` | list<string> | extensions w/o dot | yes for file-based extractors | File types this extractor handles; must be ⊆ `allowed_file_formats`. | per extractor |

**Required extractors (closed set — all 9 must be present and `enabled: true`):**
`xlsx_text`, `xlsx_image_manifest` (manifest only, **no OCR**), `csv`, `markdown`, `json_yaml`, `frontend_static`, `backend_static`, `procedure_sql`.
> Note: the brief lists 9 required slots; `json/yaml` is one extractor (`json_yaml`) handling both `.json` and `.yaml/.yml`. The single new required runtime dep is **openpyxl** (for `xlsx_text`); see `v0.2_EXTRACTOR_STRATEGY.md`.

```yaml
  extractors:
    required:
      - { name: xlsx_text,           enabled: true, extensions: [xlsx] }
      - { name: xlsx_image_manifest, enabled: true, extensions: [xlsx] }   # manifest only, NO OCR
      - { name: csv,                 enabled: true, extensions: [csv] }
      - { name: markdown,            enabled: true, extensions: [md] }
      - { name: json_yaml,           enabled: true, extensions: [json, yaml, yml] }
      - { name: frontend_static,     enabled: true, extensions: [] }       # static source scan
      - { name: backend_static,      enabled: true, extensions: [] }       # static source scan
      - { name: procedure_sql,       enabled: true, extensions: [sql] }
    optional:
      - { name: pandas,     enabled: false, extensions: [xlsx, csv] }
      - { name: markitdown, enabled: false, extensions: [xlsx, md] }
    future:
      - { name: docling,      enabled: false, extensions: [] }
      - { name: unstructured, enabled: false, extensions: [] }
```

### 4.10 `frontend_override` (global)

A global switch in addition to the per-aspect flags in §4.2. It answers the brief's requirement: *whether frontend may override screen specs or backend evidence* at the project level.

| Field | Type | Allowed values | Required | Meaning | Default |
|---|---|---|---|---|---|
| `frontend_override` | map | — | yes | Project-global frontend override stance. | (below) |
| `frontend_override.allowed_global` | boolean | `true`/`false` | yes | Global master switch. When `false`, frontend may never override anything, regardless of per-aspect flags. | `false` |
| `frontend_override.may_override_screen_spec` | boolean | `true`/`false` | yes | May frontend overrule screen specs? | `false` |
| `frontend_override.may_override_backend` | boolean | `true`/`false` | yes | May frontend overrule backend evidence? | `false` |

**Relationship to per-aspect flags (effective rule):** frontend override is permitted for an aspect **iff** `frontend_override.allowed_global == true` **AND** `source_of_truth.<aspect>.frontend_override_allowed == true`. The global switch can only *tighten* — it can never enable an aspect that set `false`. (This is the machine encoding of the project-level `frontend_override_forbidden` flag referenced in `v0.2_SPEC.md`: it is `true` exactly when `allowed_global == false`.)

## 5. `module_policy.yaml` — overrides, additions, and hard limits

`module_policy.yaml` uses the **same `qa_policy:` schema** as the project policy but every key is **optional**; present keys override, absent keys inherit (§2).

**A module MAY:**
- Refine `source_of_truth.<aspect>.primary/secondary` ordering for that module (e.g. a module where `procedure` is more authoritative).
- Add module-specific `forbidden_assumptions` (merged/unioned with project list).
- Add steps to `defer_until_frontend_verified` (merged/unioned).
- Override `module_scope.module_path_template` / `spec_glob` and add module-local `repository_path_mapping` entries (e.g. the ORD module's exact extracted folder).
- Tighten any boolean toward the **safer** value (e.g. set `ocr_image.usable: false`, raise `needs_human_confirmation` requirements, set an aspect's `frontend_override_allowed: false` even if the project allowed it).
- Disable an optional extractor it does not need.

**A module MUST NOT (validation error if violated):**
- Set any `frontend_override_allowed` / `frontend_override.*` to `true` when the project set it `false`. **Rule: a module can only forbid more, never grant more.** Granting frontend override where the project forbids it is a hard validation error.
- Re-enable OCR `usable: true` or raise `ocr_image.max_confidence` above the project cap.
- Set `ocr_image.needs_human_confirmation: false` (it must stay `true`).
- Remove entries from project-level `forbidden_assumptions` or `defer_until_frontend_verified` (lists are additive only; deletion is rejected).
- Disable a **required** extractor.
- Change `review_strategy` to a less-scoped value (e.g. `full_repo`) — module policy is module-scoped by definition.

Formal monotonicity rule (used by the validator): for every boolean permission `P` (any `frontend_override*`, `ocr_image.usable`), the effective module value must satisfy `module_P ⇒ project_P` (module may turn a permission OFF, never ON). For every confidence cap, `module_cap ≤ project_cap` in the order `low < medium < high`.

## 6. Behavior contract (how policy drives commands)

### 6.1 Load order (every command)
Before any QA work, each command performs, in order:
1. Resolve `qa-context/project_policy.yaml`. If missing → §6.2.
2. Parse + validate it (§7). On validation error → halt non-zero (exit-code contract in `v0.2_SPEC.md`).
3. Read `project_memory.md` / `project_assumptions.yaml` (see `v0.2_PROJECT_MEMORY_SPEC.md`).
4. If a `--module` is given: load `qa-context/modules/{MODULE}/module_policy.yaml`, validate it, and compute the **effective policy** by merging onto the project policy (§2 precedence, §5 monotonicity).
5. Read `module_memory.md`.
These five reads correspond exactly to the first steps of `validation_order` (`read_project_policy`, `read_project_memory`, `read_module_policy`, `read_module_memory`).

### 6.2 Missing-policy behavior
If `project_policy.yaml` is absent, the toolkit does **one** of:
- **Halt** with a clear non-zero error instructing the user to run `erpqa policy-init`; **or**
- **Create a DRAFT policy** (scaffold from the pilot strict defaults) marked `draft: true` and `needs_human_confirmation: true`, then **halt before any QA work** until a human confirms. `human_confirmation_requirements.require_for_missing_policy_draft` MUST be `true`. A DRAFT policy never silently proceeds into auditing.
(`policy-init` itself is the supported way to create a non-draft policy; see `v0.2_SPEC.md` §2.1.)

### 6.3 How `validation_order` + defer list are consumed by module-audit
The module-audit pipeline executes steps **strictly in `validation_order`**. Any step also listed in `defer_until_frontend_verified` is **scheduled but not run as authoritative** until the frontend has been verified for the module (frontend mismatches resolved / human-confirmed). Deferred steps may still be *previewed* (e.g. `procedure_mapping_report.md` is generated but flagged DEFERRED — see `v0.2_SPEC.md` §6) but their findings are non-authoritative and carry `needs_human_confirmation: true`.

### 6.4 How `source_of_truth` precedence feeds the comparison engine
For each compared aspect, the comparison engine (logic in `v0.2_SPEC.md` §5) treats `primary` sources as authoritative in listed order and `secondary` sources as corroboration only. A frontend value that conflicts with primary is emitted as a **mismatch finding**, never a silent fix. Frontend is allowed to override **only** when the effective-policy rule of §4.10 yields `true` for that aspect; otherwise `frontend_override_forbidden` is set on the finding and the conflict is reported for human decision.

### 6.5 How `extractors` config gates the adapter registry
The adapter registry (see `v0.2_ARCHITECTURE.md`) is populated from `extractors`:
- A **required** extractor that is missing/disabled → fatal config error (required extractors must be real + wired + tested).
- An **optional** extractor runs only if `enabled: true` **and** its dependency imports successfully (lazy import); otherwise it is silently skipped.
- A **future** extractor is never auto-enabled in v0.2.
- A file is dispatched to an extractor only if its extension is in that extractor's `extensions` **and** in `allowed_file_formats`. Files outside `allowed_file_formats` are ignored with a logged note.

### 6.6 How the OCR/image confidence cap is enforced
Any item whose `source`/`extraction_method` indicates image-derived or OCR origin has its confidence **clamped to `extraction_confidence_rules.ocr_image.max_confidence`** (≤ `medium`) and `needs_human_confirmation` forced to `true`. A `high` assigned to such an item is downgraded and flagged. If `ocr_image.usable == false`, image-derived text is dropped entirely (manifest-only behavior). This clamp happens before any comparison so capped confidence propagates per `v0.2_SPEC.md` §5.

## 7. Validation rules for the policy file itself

The validator (invoked at load and by `policy-init`) enforces:

**Required keys (project policy):** `qa_policy.name`, `review_strategy`, `source_of_truth` (all 5 aspects, each with `primary` + `secondary` + `frontend_override_allowed`), `validation_order`, `defer_until_frontend_verified`, `forbidden_assumptions`, `allowed_file_formats`, `extraction_confidence_rules` (incl. `ocr_image.{usable,max_confidence,needs_human_confirmation}`), `human_confirmation_requirements`, `module_scope`, `repository_path_mapping` (≥1), `extractors` (with `required` tier present), `frontend_override`.

**Enum / type checks:**
- `review_strategy` ∈ {`module_by_module`} (v0.2).
- Every source token ∈ {`screen_spec`, `backend`, `procedure`, `frontend`}; no token in both `primary` and `secondary` of one aspect; `primary` non-empty.
- Every `validation_order` / `defer_until_frontend_verified` entry ∈ step vocabulary (§4.3); defer ⊆ validation_order; the four read-steps lead `validation_order` in order.
- `allowed_file_formats` lowercase, no leading dot.
- Every `frontend_override_allowed` and every `frontend_override.*` value **MUST be boolean** — strings like `"false"` are a validation error.
- `extraction_confidence_rules.ocr_image.max_confidence` ∈ {`low`, `medium`} (a value of `high` is a **validation error**); `ocr_image.needs_human_confirmation` MUST be `true`.
- All 9 required extractors present and `enabled: true`; each extractor's `extensions` ⊆ `allowed_file_formats`.

**Cross-file / monotonicity checks (module policy):**
- A module policy that sets any `frontend_override_allowed` or `frontend_override.*` to `true` while the **project** sets it `false` → **validation error** ("module cannot grant frontend override forbidden by project").
- A module `ocr_image.max_confidence` above the project cap, or `ocr_image.usable: true` over a project `false`, or `ocr_image.needs_human_confirmation: false` → validation error.
- A module attempting to remove a project `forbidden_assumptions` / defer entry, or disable a required extractor, or set `review_strategy` looser → validation error.

**Draft policies:** if `draft: true`, validation passes structurally but the command halts pending human confirmation (§6.2).

### 8b. A hypothetical laxer project (frontend may override `visible_text` only)

This shows the schema generalizes to projects that *do* trust their frontend for some aspects. Only the changed keys are shown.

```yaml
qa_policy:
  name: lenient_frontend_text_trusted
  review_strategy: module_by_module
  source_of_truth:
    visible_text:    { primary: [frontend, screen_spec], secondary: [backend], frontend_override_allowed: true }  # frontend leads & may override
    grid_columns:    { primary: [screen_spec, backend],  secondary: [frontend], frontend_override_allowed: false }
    form_fields:     { primary: [screen_spec, backend],  secondary: [frontend], frontend_override_allowed: false }

    procedure_mapping:{ primary: [backend, procedure],   secondary: [frontend, screen_spec], frontend_override_allowed: false }
  frontend_override:
    allowed_global: true              # master switch ON so an aspect-level true can take effect
    may_override_screen_spec: true    # but only where an aspect allows it (visible_text)
    may_override_backend: false
  # ...all other blocks as in the strict pilot...
```

**Effective behavior:** for `visible_text`, a frontend label that differs from the screen spec is accepted as authoritative (no mismatch raised); for every other aspect frontend still cannot override and conflicts are still reported. A module under this project could **tighten** `visible_text` back to `frontend_override_allowed: false`, but could **not** further loosen any other aspect (§5/§7).
