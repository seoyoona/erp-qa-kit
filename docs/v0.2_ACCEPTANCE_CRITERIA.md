# ERP QA Kit v0.2 — Acceptance Criteria (Definition of Done)

> **Status:** Planning only. This is the **testable definition-of-done** for ERP QA Kit **v0.2**, intended to be read and executed by Codex during implementation. It contains **no implementation code** and modifies **no v0.1 file**.
>
> **How to read this:** every criterion is an objectively verifiable checkbox. Where a criterion is testable from a shell, the exact command and the expected result are given. `Given/When/Then` is used where behavior depends on state. Sibling v0.2 documents are referenced by name (see end) and **must not be duplicated** here.
>
> **Conventions used below**
> - `ROOT` = repository root (`/Users/yoona/Documents/erp-qa-kit`).
> - `PROJ` = a target-project path containing `qa-context/` (the positional `project_path`).
> - `MOD` = a module name passed via `--module <NAME>`.
> - "exit 0" / "non-zero exit" refer to the process exit code (`echo $?`).
> - "clear error, not traceback" means a single human-readable line on stderr **and** the process never prints a Python traceback for an expected/handled error condition.
> - Confidence rules referenced throughout are defined in `docs/v0.2_EXTRACTOR_STRATEGY.md` and summarized in `docs/v0.2_PLAN.md §5`.

## A. Backward Compatibility

- [ ] **A1.** All three v0.1 commands still exist and run unchanged: `init`, `validate`, `generate-sql`.
  `erpqa init <PROJ>` exits 0 and creates `qa-context/`; `erpqa validate <PROJ>` exits 0 on valid artifacts; `erpqa generate-sql <PROJ>` exits 0. Each still takes a **positional** `project_path`.
- [ ] **A2.** The v0.1 test suite still passes with **19** tests collected and green.
  `cd ROOT && python -m pytest tests/test_validation.py tests/test_end_to_end.py tests/test_sql_safety.py tests/test_write_containment.py tests/test_no_db.py -q` → exit 0, `19 passed`.
- [ ] **A3.** No v0.1 source file is modified by v0.2. `git diff --name-only v0.1.0..HEAD` shows v0.1 files (`erpqa/cli.py`, `erpqa/core/paths.py`, `erpqa/core/constants.py`, existing reporters/generators/adapters) only as **additive** changes (new functions/registrations) — v0.1 behavior, signatures, and outputs are byte-stable where exercised by the 19 tests.
- [ ] **A4.** Dependency surface is additive: the only new **required** runtime dependency is `openpyxl`. `pyproject.toml [project].dependencies` contains exactly `PyYAML` and `openpyxl`; `requires-python` remains `>=3.11`.
  `Given` a fresh venv with only `PyYAML` + `openpyxl` installed, `When` any required v0.2 command runs, `Then` it succeeds (no missing-import error).
- [ ] **A5.** The `erpqa` console entry point is unchanged (`erpqa = "erpqa.cli:main"`); v0.2 commands are added as new subparsers, not a new binary.

## B. CLI Surface

- [ ] **B1.** All **10** v0.2 subcommands exist: `policy-init`, `module-init`, `extract-spec`, `extract-frontend`, `extract-backend`, `extract-procedure`, `compare-contract`, `module-report`, `module-handoff`, `module-audit`.
  `erpqa --help` lists all 10 (plus the v0.1 three) and exits 0.
- [ ] **B2.** Each subcommand supports `--help` with exit 0: for `C in {the 10 above}`, `erpqa $C --help; echo $?` → `0`, and the help text names its positional/optional arguments.
- [ ] **B3.** `policy-init` is **project-level**: it accepts positional `project_path` and does **not** require `--module`. The other 9 v0.2 commands accept positional `project_path` **and** `--module <NAME>`.
- [ ] **B4.** Missing/invalid `project_path` → clear non-zero error, not a traceback.
  `erpqa module-init /no/such/path --module M; echo $?` → non-zero, stderr is one readable line (e.g. "project_path does not exist: …"), no traceback.
- [ ] **B5.** Missing required `--module` on a module command → clear non-zero error.
  `erpqa extract-spec <PROJ>; echo $?` → non-zero with a readable "missing --module" message (argparse usage line acceptable; no traceback).
- [ ] **B6.** Invalid/unknown module (no `modules/<MOD>/` scaffold and command requires it) → clear non-zero error naming the module, not a traceback.

## C. policy-init

- [ ] **C1.** On a project with no policy, `erpqa policy-init <PROJ>` exits 0 and creates all three: `qa-context/project_policy.yaml` (machine-readable, valid YAML), `qa-context/project_memory.md` (PM-readable Markdown), `qa-context/project_assumptions.yaml` (valid YAML). Schemas per `docs/v0.2_PROJECT_POLICY_SPEC.md` and `docs/v0.2_PROJECT_MEMORY_SPEC.md`.
- [ ] **C2.** `project_policy.yaml` contains at minimum the keys: `frontend_override_allowed` (default `false`), `validation_order`, `defer_until_frontend_verified` (default `true`), and is parseable by PyYAML.
- [ ] **C3.** No-clobber: `Given` the three files already exist, `When` `erpqa policy-init <PROJ>` runs again, `Then` it does **not** overwrite them (existing content preserved) and reports that they already exist; exit 0. Verify by capturing checksums before/after — they are unchanged.
- [ ] **C4.** Draft requires human confirmation: a freshly created policy is marked as a draft requiring confirmation (e.g. `human_confirmed: false` plus a visible "REVIEW REQUIRED" marker in `project_memory.md`/assumptions). No command treats an unconfirmed draft as authoritative without surfacing the unconfirmed state.

## D. Policy / Memory Enforcement

- [ ] **D1.** Every v0.2 command reads project policy + memory (and module policy + memory when module-scoped) **before** doing any work. Verifiable: a test stubs/intercepts the policy+memory readers and asserts they are invoked before any extractor/comparison/report logic for each of the 9 module commands and for orchestration.
- [ ] **D2.** Missing `project_policy.yaml` halts or drafts: `Given` `qa-context/` exists but `project_policy.yaml` is absent, `When` any module command runs, `Then` it either (a) stops with a clear non-zero error instructing the user to run `policy-init`, **or** (b) creates a human-confirmation **draft** policy and refuses to proceed as if confirmed. It never silently proceeds with hidden defaults.
- [ ] **D3.** Reports and handoffs explicitly state both memories were read: every generated report under `reports/` and every file under `handoff/` contains an explicit line confirming `project_memory.md` was read and (when present) `module_memory.md` was read — e.g. `project_memory_read: true` / `module_memory_read: true` fields or an equivalent prose statement naming both files.
- [ ] **D4.** Module memory overrides/refines project memory — testable example.
  `Given` `project_memory.md` states fact X and `modules/<MOD>/module_memory.md` overrides X with X', `When` a module command runs for `<MOD>`, `Then` the effective value used and cited is X' (override), and the report records that the module-level value took precedence. A fixture + test encodes this exact case.
- [ ] **D5.** Frontend-vs-policy conflicts are **reported, never resolved**: when `frontend_override_allowed: false`, any finding where the frontend would override the screen spec/backend is emitted as a mismatch (with `frontend_override_forbidden: true`) rather than silently reconciled.

## E. module-init

- [ ] **E1.** `erpqa module-init <PROJ> --module <MOD>` exits 0 and scaffolds the full tree under `qa-context/modules/<MOD>/`: `module_manifest.yaml`, `module_policy.yaml`, `module_memory.md`, and the empty-but-present contract targets (`screen_contract.yaml`, `frontend_contract.yaml`, `backend_contract.yaml`, `procedure_contract.yaml`) plus `reports/` and `handoff/` directories. Layout matches `docs/v0.2_ARCHITECTURE.md`.
- [ ] **E2.** Idempotent + no-clobber: running `module-init` twice for the same `<MOD>` does not overwrite existing files; exit 0 both times. Verify by checksum stability of any file that already had content.
- [ ] **E3.** `module_manifest.yaml` records the module's source/extracted-folder paths (so a module-scoped extracted folder can be located without a full repo).
- [ ] **E4.** `module_policy.yaml` and `module_memory.md` are valid and clearly documented as **refining/overriding** their project-level equivalents.

## F. Required Extractors

> One verifiable block per REQUIRED adapter. Each block requires: (i) the adapter runs on a fixture file of its declared type, (ii) it produces **normalized** output (per `docs/v0.2_EXTRACTOR_STRATEGY.md`), (iii) it assigns the **correct default confidence + `needs_human_confirmation`** per the confidence rules, and (iv) it is covered by a test/fixture under `tests/`.

- [ ] **F0. Not placeholder-only (global).** For every REQUIRED adapter, a test asserts **real, non-empty, structurally-correct** output on its fixture (extracted field names/values present), failing if the adapter returns empty/stub/`NotImplemented`. The extractor registry lists all 8 REQUIRED adapters and is wired to the CLI extract commands.
- [ ] **F1. `xlsx_text_adapter` (openpyxl).** Runs on a fixture `.xlsx`, extracts cell text into normalized rows/fields; default confidence **medium**, `needs_human_confirmation: true`. Test reads a committed fixture `.xlsx` and asserts expected cell text appears.
- [ ] **F2. `xlsx_image_manifest_adapter` (openpyxl).** Runs on a fixture `.xlsx` containing ≥1 embedded image; lists each embedded image with its anchor/position; performs **NO OCR**. Output confidence is **low or medium** (never high) with `needs_human_confirmation: true`. Test asserts the manifest enumerates the embedded image(s) and that no image text/content was decoded.
- [ ] **F3. `csv_adapter` (stdlib `csv`).** Runs on a fixture `.csv` using only the standard library; normalized output; default confidence **medium**, `needs_human_confirmation: true`. Test asserts header + row extraction.
- [ ] **F4. `markdown_adapter` (stdlib).** Runs on a fixture `.md` using only the standard library; normalized output; default confidence **medium**, `needs_human_confirmation: true`.
- [ ] **F5. `json_yaml_adapter` (stdlib `json` + PyYAML).** Runs on fixture `.json` and `.yaml`; for **explicitly machine-readable** spec input, confidence **may be high** (and `needs_human_confirmation` may be `false`); otherwise medium + needs confirm. Test covers both a `.json` and a `.yaml` fixture.
- [ ] **F6. `frontend_static_adapter`.** Deterministically extracts frontend fields/labels/columns/buttons from a fixture frontend source folder; default confidence **medium/low**; inferred mappings → `needs_human_confirmation: true`. Test asserts deterministic, repeatable output (same input → same output).
- [ ] **F7. `backend_static_adapter`.** Deterministically extracts backend evidence (fields/endpoints) from a fixture backend source folder; default confidence **medium/low**, `needs_human_confirmation` true for inferred mappings.
- [ ] **F8. `procedure_sql_adapter` (NO DB exec).** Parses fixture SQL/procedure **text** deterministically; default confidence **medium/low**, needs confirm. A test asserts that **no database connection or query execution** occurs (no DB driver import/connect; reuses/honors the v0.1 no-DB guarantee — see `docs/v0.2_CONSTRAINTS.md`).
- [ ] **F9. Confidence cap enforced.** A test asserts that no required adapter ever emits `confidence: high` for image-/OCR-derived data, and that image-derived/uncertain outputs always carry `needs_human_confirmation: true`.

## G. Optional Extractors

- [ ] **G1.** `pandas` and `markitdown` are OPTIONAL: imported lazily; their absence does **not** break any required flow.
  `Given` a venv with only `PyYAML` + `openpyxl` (no pandas, no markitdown), `When` every required extract command and `module-audit` run on a fixture module, `Then` all succeed (exit 0) and any optional-only feature degrades gracefully with a clear "optional dependency not installed" notice rather than an error.
- [ ] **G2.** When uninstalled, the kit still runs all 8 required adapters (covered by the F-block tests running in the minimal venv).
- [ ] **G3.** `markitdown_adapter` / any image-derived output is **low or medium** confidence and **always** `needs_human_confirmation: true`, **never** high. A test (skipped if markitdown absent, but asserting the contract when present) verifies this cap.
- [ ] **G4.** Docling and Unstructured are **FUTURE-only**: documented in `docs/v0.2_EXTRACTOR_STRATEGY.md`, **not** implemented, not imported, and not in `dependencies`.

## H. Contracts & Validation

- [ ] **H1.** Each of `screen_contract.yaml`, `frontend_contract.yaml`, `backend_contract.yaml`, `procedure_contract.yaml` validates against its schema (per `docs/v0.2_SPEC.md`).
- [ ] **H2.** Every contract **item** carries the provenance trio: `source`, `confidence`, `needs_human_confirmation`. A test rejects any item missing one of the trio.
- [ ] **H3.** Missing-required-field → validation error naming **file + field**, non-zero exit.
  `Given` a contract fixture with a required field removed, `When` `erpqa validate <PROJ>` (extended) or the contract validator runs, `Then` it exits non-zero and prints a message naming the offending file and the missing field (no traceback).
- [ ] **H4.** Valid contracts pass: on a clean fixture module, contract validation exits 0.

## I. Comparison Engine

> `erpqa compare-contract <PROJ> --module <MOD>` compares the frontend contract against the source of truth (screen spec + backend evidence). **One verifiable check per mismatch type**, each backed by a fixture pair where the type is the only injected difference.

- [ ] **I1.** Missing FE columns (in source of truth, absent in frontend) → detected.
- [ ] **I2.** Unexpected FE columns (in frontend, absent in source of truth) → detected.
- [ ] **I3.** Label mismatch → detected.
- [ ] **I4.** Order mismatch → detected.
- [ ] **I5.** Visibility mismatch → detected.
- [ ] **I6.** Required/optional mismatch → detected.
- [ ] **I7.** Readonly/disabled mismatch → detected.
- [ ] **I8.** Field-key mismatch → detected.
- [ ] **I9.** Search-filter mismatch → detected.
- [ ] **I10.** Grid-column mismatch → detected.
- [ ] **I11.** Form-field mismatch → detected.
- [ ] **I12.** Buttons/actions mismatch → detected.
- [ ] **I13.** Internal/audit/system fields exposed when not allowed → detected.
- [ ] **I14.** Frontend fields overriding screen-spec labels/definitions → detected **and** flagged `frontend_override_forbidden: true`.
- [ ] **I15.** Frontend never overrides the source of truth when policy forbids it (`frontend_override_allowed: false`): the engine reports the conflict and does not rewrite the screen contract.
- [ ] **I16.** `needs_human_confirmation` propagates: when **either** compared side is low confidence, the resulting mismatch record carries `needs_human_confirmation: true`. (Verified by a fixture where the frontend side is low confidence.)

## J. Reports

- [ ] **J1.** `erpqa module-report <PROJ> --module <MOD>` exits 0 and generates the module reports under `qa-context/modules/<MOD>/reports/`: `screen_frontend_mismatch_report.md`, `column_mismatch_report.md`, `frontend_contract_report.md`, `procedure_mapping_report.md`.
- [ ] **J2.** The mismatch report includes **every** required field for each mismatch entry: `screen_id`, `module`, `mismatch_type`, `expected_from_source_of_truth`, `actual_frontend`, `source_files`, `confidence`, `needs_human_confirmation`, `severity`, `suggested_fix_type`, `frontend_override_forbidden`, `backend_evidence_agrees_or_conflicts`, `project_memory_read`, `module_memory_read`, `handoff_ready_ai_fix_instruction`. A test asserts all 15 fields are present per entry.
- [ ] **J3.** Reports are PM-readable Markdown (headings/tables, no raw stack traces or code-only dumps) and cite the memory reads per **D3**.

## K. Handoff

- [ ] **K1.** `erpqa module-handoff <PROJ> --module <MOD>` exits 0 and generates `qa-context/modules/<MOD>/handoff/frontend_fix_handoff.md` and `codex_frontend_fix_prompt.md`.
- [ ] **K2.** Each handoff includes: exact screen IDs; expected vs actual; likely FE files/components; related API/backend evidence; a **frontend-override-forbidden** reminder; confirmation that memory was read; and post-fix validation steps (commands to re-run).
- [ ] **K3.** Handoffs instruct fixing the **TARGET FRONTEND repo**, not the ERP QA Kit, and explicitly state the kit performs **no auto-modification** of target code.

## L. module-audit Orchestrator

- [ ] **L1.** `erpqa module-audit <PROJ> --module <MOD>` runs the stages in the policy's `validation_order` (extract-spec → extract-frontend → extract-backend → extract-procedure → compare-contract → module-report → module-handoff, as ordered by policy).
- [ ] **L2.** Honors `defer_until_frontend_verified`: `Given` `defer_until_frontend_verified: true`, `When` `module-audit` runs, `Then` procedure/business-logic validation stages are **skipped** (extraction may still occur but procedure *validation* is deferred), and the run records that they were deferred.
- [ ] **L3.** Produces all expected module artifacts (contracts, four reports, two handoff docs) for a clean fixture module.
- [ ] **L4.** Exits **0** on a clean fixture module.
  `erpqa module-audit <PROJ> --module <FIXTURE_MOD>; echo $?` → `0`.

## M. Module-scope & Safety

- [ ] **M1.** Runs on a module-specific **extracted folder** referenced by `module_manifest.yaml` — no full repository present — and still completes (a fixture module ships only an extracted subfolder).
- [ ] **M2.** Does not require cloning all repos: a test asserts the fixture has only one module's extracted folder and `module-audit` still exits 0.
- [ ] **M3.** Never writes outside `qa-context/`: reuses v0.1's `assert_within_qa_context` guard; a test attempts a write to a path outside `qa-context/` during a v0.2 flow and asserts it is refused (extends `tests/test_write_containment.py`).
- [ ] **M4.** Never executes DB: a test asserts no DB driver is imported/connected and no query executed across v0.2 flows (extends `tests/test_no_db.py`).
- [ ] **M5.** Never modifies target ERP code: a test asserts that running any v0.2 command leaves the target source/extracted folder byte-for-byte unchanged (checksum the source tree before/after).
