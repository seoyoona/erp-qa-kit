# ERP QA Kit v0.2 — Project Memory Specification

> **Status:** Planning / authoritative spec. PLANNING ONLY — this document defines behavior for Codex to implement in v0.2. It does not contain code.
> **Audience:** Codex (implementer) and the non-developer PM who operates the toolkit.
> **Scope:** Persistent project memory, module memory, and `project_assumptions.yaml`.
> **Siblings (do not duplicate):** `docs/v0.2_PROJECT_POLICY_SPEC.md` (enforceable machine policy), `docs/v0.2_SPEC.md` (overall v0.2 behavior), `docs/v0.2_DRY_RUN_GUIDE.md` (how a PM runs a dry run), `docs/v0.2_ARCHITECTURE.md` (component layout). This document references those by name and does not restate their contents.

---

## 1. Purpose & the Problem It Solves

The operator of this toolkit is a **non-developer PM** who runs the QA kit **repeatedly** — across multiple ERP modules and across multiple project directories, over many sessions.

The core problem: **if project-specific rules live only in the chat prompt, agents forget them.** Each new run starts cold, re-derives assumptions, and produces QA results based on the *wrong* assumptions (for example, treating the frontend as the source of truth, or trusting backend implementation that the PM knows is not yet verified). This wastes the PM's time and silently corrupts QA findings.

The solution is **persistent, file-based, PM-readable memory that is read before every run**:

- **Persistent** — memory lives in files inside the project (`qa-context/`), not in prompt history. It survives across sessions, machines, and runs.
- **File-based** — every command and skill loads it from disk at the start of work (the *read-before-work contract*, Section 6).
- **PM-readable** — `project_memory.md` and `module_memory.md` are written as plain prose/bullets the PM can read and edit without developer help.
- **Machine-readable companion** — `project_assumptions.yaml` carries the same knowledge in a structured form agents can reason over (status, confidence, `needs_human_confirmation`).

The design goal: **an agent should never QA a module with a wrong assumption that the PM has already corrected in a previous run.** Memory is how a correction made once becomes permanent.

This document covers **memory** (human narrative + assumptions). The enforceable machine rules (the *policy*) are a separate concern specified in `docs/v0.2_PROJECT_POLICY_SPEC.md`. See Section 7 for how they interact.

## 2. File Set & Roles

All memory files live under the project's `qa-context/` directory. The architecture of `qa-context/` as a whole is described in `docs/v0.2_ARCHITECTURE.md`; here we define only the memory files.

```
qa-context/
├── project_memory.md            # PM-readable prose: project-wide narrative & rules
├── project_assumptions.yaml     # Machine-readable assumptions ledger
├── project_policy.yaml          # Enforceable machine rules — see v0.2_PROJECT_POLICY_SPEC.md
└── modules/
    ├── ORD/
    │   └── module_memory.md      # PM-readable prose: ORD-specific refinements/overrides
    └── {MODULE}/
        └── module_memory.md      # one per module under QA
```

| File | Location | Readable by | Role |
|------|----------|-------------|------|
| `project_memory.md` | `qa-context/` | PM (prose) | The human narrative of project-wide rules, trust levels, source-of-truth decisions, validation order, and open questions. The authoritative *story* of how this ERP must be QA'd. |
| `project_assumptions.yaml` | `qa-context/` | Machine (PM-editable) | Structured ledger of discrete assumptions, each with status, confidence, source, and a human-confirmation flag. Agents reason over this; the PM promotes/rejects entries across runs. |
| `module_memory.md` | `qa-context/modules/{MODULE}/` | PM (prose) | Module-specific narrative that **refines or overrides** `project_memory.md` for one module only. Where a module differs from the project default, that difference is recorded here. |
| `project_policy.yaml` | `qa-context/` | Machine | **Not specified here.** Enforceable rules. See `docs/v0.2_PROJECT_POLICY_SPEC.md`. |

**Roles in one line each:**

- `project_memory.md` — *PM-readable prose; project-wide.*
- `project_assumptions.yaml` — *machine-readable; the assumptions ledger.*
- `module_memory.md` — *PM-readable prose; module-specific; refines/overrides project memory.*

## 3. `project_memory.md` Format

### 3.1 Recommended PM-friendly structure

`project_memory.md` is written for the PM. It is prose and bullets, organized into stable sections so the PM always knows where to add a new finding. Recommended sections (in order):

```markdown
# Project Memory — {Project Name}

_Last updated: {YYYY-MM-DD} by {name}. Read this before every run._

## Project Overview
Short description of the ERP, who the client is, and the current state of QA.

## Repositories & Scope
Which repos exist, which are in scope, how code is obtained (extracted folders vs clone),
and what "a module" means here.

## Source-of-Truth Rules
What is authoritative for visible text, labels, columns, filters, and form fields —
and what is explicitly NOT authoritative.

## Trust Levels
How much each artifact is trusted (screen spec, backend, stored procedures, frontend).

## Validation Order & Deferrals
The required sequence of validation and what is deferred until earlier steps pass.

## Open Questions
Things not yet decided; items awaiting human confirmation. Mirror these into
project_assumptions.yaml where they are discrete assumptions.
```

The PM may add notes freely under these headings. Agents read the whole file as narrative context and must honor it; where a statement is a discrete, testable claim, it should also exist as an entry in `project_assumptions.yaml` (Section 5) so it can carry status/confidence.

### 3.2 Default pilot seed content (verbatim)

When memory is first created (see Section 6 on `policy-init` / `module-init`), `project_memory.md` is **seeded with exactly the following content**, reproduced verbatim as readable bullets. This is the default pilot memory:

```markdown
# Project Memory — ERP QA Kit Pilot

_Last updated: SEED. Read this before every run._

## Project Overview
- The ERP has 7 modules but only one module has been fully verified so far.

## Repositories & Scope
- There are 3 repositories total, including ORD.
- The full repositories are too large to clone/analyze all at once.
- Module-level extracted folders are preferred.

## Source-of-Truth Rules
- Frontend override is forbidden.
- Screen specification and backend evidence are the source of truth for visible text, labels, columns, search filters, and form fields.
- Frontend is the implementation to validate, not the source of truth.

## Trust Levels
- Backend implementation is not fully trusted yet.
- Stored procedures were provided by the client and may be closer to intended business logic, but integration is not fully verified.

## Validation Order & Deferrals
- Frontend screen contract validation must happen before procedure/business logic validation.
- Procedure/business logic validation is deferred until frontend mismatches are resolved.

## Open Questions
- (Add items here as they arise during dry runs.)
```

> The eleven seed statements above (across the Overview, Repositories, Source-of-Truth, Trust, and Validation sections) are the canonical pilot memory. They MUST appear verbatim in the seed file.

## 4. `module_memory.md` Format

### 4.1 Structure

`module_memory.md` is created per module under `qa-context/modules/{MODULE}/`. It mirrors the project memory sections but contains **only what differs for this module**. Recommended structure:

```markdown
# Module Memory — {MODULE}

_Last updated: {YYYY-MM-DD} by {name}. Read this AFTER project_memory.md; this file wins on conflict._

## Module Overview
What this module is, its verification status, where its extracted folder lives.

## Refinements (adds detail to project memory)
Notes that ADD specificity without contradicting project memory.

## Overrides (replaces project memory for this module)
Notes that DEVIATE from project memory. Each override must state which project rule it replaces and why.

## Module Open Questions
Module-specific uncertainties; mirror discrete ones into project_assumptions.yaml as applies_to: module:{MODULE}.
```

### 4.2 Refine vs Override — semantics

- **Refine** — the module note is *consistent* with project memory and adds detail/specificity. Project memory still applies; the module note narrows or elaborates it. No conflict.
- **Override** — the module note *contradicts* a project-memory statement for this module only. The module note **wins for this module**. The override must explicitly name the project rule it replaces, so the deviation is visible and auditable.

**Conflict resolution rule:** when project memory and module memory disagree, **module memory overrides project memory** for that module. (This is distinct from memory-vs-policy conflicts; see Section 7.)

### 4.3 Worked ORD example

```markdown
# Module Memory — ORD

_Last updated: 2026-06-14 by yoona. Read this AFTER project_memory.md; this file wins on conflict._

## Module Overview
- ORD (Orders) lives in repository ORD; extracted module folder only, no full clone.
- ORD is one of the not-yet-fully-verified modules.

## Refinements (adds detail to project memory)
- Source-of-truth rule still holds: screen spec + backend evidence are authoritative for labels/columns/filters/fields.
- ORD screen specs are delivered as a mix of text specs and exported PDFs.

## Overrides (replaces project memory for this module)
- OVERRIDE of "Screen specification ... is the source of truth": for the ORD order-search screen,
  the screen spec is IMAGE-ONLY (a screenshot with no machine-readable field list). Because the
  spec cannot be parsed reliably, this screen's field/label/column contract MUST be human-confirmed
  before any mismatch is reported as a defect. Set needs_human_confirmation: true for any contract
  derived from this image-only spec.

## Module Open Questions
- Does the image-only order-search spec match the latest client wireframe? Awaiting PM confirmation.
```

In this example, the project-wide rule "screen spec is the source of truth" is *refined* in general (text + PDF) but *overridden* for one screen where the spec is image-only and therefore cannot be trusted without a human in the loop. The override names the rule it replaces and forces `needs_human_confirmation`.

## 5. `project_assumptions.yaml` Schema

`project_assumptions.yaml` is the machine-readable ledger of discrete assumptions. It is a single top-level list, `assumptions:`. Each item carries the following fields:

| Field | Type | Allowed values / notes |
|-------|------|------------------------|
| `assumption_id` | string | Stable unique id, e.g. `A-001`. Never reused. |
| `statement` | string | One clear, testable claim. |
| `status` | enum | `proposed` \| `confirmed` \| `rejected` |
| `applies_to` | string | `project` \| `module:{NAME}` (e.g. `module:ORD`) |
| `source` | string | Where it came from: `seed`, `screen_spec`, `backend_evidence`, `client_statement`, `dry_run:{date}`, `human:{name}`, etc. |
| `confidence` | enum/number | `low` \| `medium` \| `high` (or 0.0–1.0). How sure the agent is. |
| `needs_human_confirmation` | boolean | `true` if the assumption must be confirmed by a human before it is relied upon for defect reporting. |

### 5.1 Example (seeded from the pilot memory)

```yaml
# qa-context/project_assumptions.yaml
assumptions:
  - assumption_id: A-001
    statement: "Backend implementation is not fully trusted yet."
    status: confirmed
    applies_to: project
    source: seed
    confidence: high
    needs_human_confirmation: false

  - assumption_id: A-002
    statement: "Stored procedures provided by the client reflect the intended business logic."
    status: proposed
    applies_to: project
    source: client_statement
    confidence: medium
    needs_human_confirmation: true

  - assumption_id: A-003
    statement: "Screen specification and backend evidence are the source of truth for visible text, labels, columns, search filters, and form fields."
    status: confirmed
    applies_to: project
    source: seed
    confidence: high
    needs_human_confirmation: false

  - assumption_id: A-004
    statement: "Frontend is the implementation under test and must never be treated as the source of truth (override forbidden)."
    status: confirmed
    applies_to: project
    source: seed
    confidence: high
    needs_human_confirmation: false

  - assumption_id: A-005
    statement: "Procedure/business-logic validation is deferred until frontend screen-contract mismatches are resolved."
    status: confirmed
    applies_to: project
    source: seed
    confidence: high
    needs_human_confirmation: false

  - assumption_id: A-006
    statement: "The ORD order-search screen spec is image-only and its field/label contract must be human-confirmed before reporting defects."
    status: proposed
    applies_to: module:ORD
    source: dry_run:2026-06-14
    confidence: medium
    needs_human_confirmation: true
```

### 5.2 Promotion lifecycle: proposed → confirmed (or rejected)

Assumptions are how uncertainty becomes durable knowledge across runs:

1. An agent (or PM) records a new claim as `status: proposed`, usually with `needs_human_confirmation: true`, citing its `source` and `confidence`.
2. During or after a **dry run** (see `docs/v0.2_DRY_RUN_GUIDE.md`), the PM reviews proposed assumptions.
3. The PM **promotes** an assumption by editing `status: proposed` → `status: confirmed` and setting `needs_human_confirmation: false`. The PM may also bump `confidence`. This is a human act; agents must not self-promote `proposed` → `confirmed`.
4. If the claim is wrong, the PM sets `status: rejected`. Rejected assumptions are kept (not deleted) so the same wrong assumption is not silently re-proposed in a future run.
5. On subsequent runs, agents treat `confirmed` assumptions as reliable inputs and treat `rejected` ones as known-false guardrails. `proposed` assumptions are used cautiously and always surfaced for confirmation.

**Any uncertain interpretation an agent makes during work MUST be written back as a `proposed` assumption with `needs_human_confirmation: true`** rather than acted on as if confirmed.

## 6. Read-Before-Work Contract

Every command and skill in the toolkit MUST, **before doing any QA work**:

1. **Load** `qa-context/project_memory.md` and `qa-context/project_assumptions.yaml`.
2. If a module is in scope, **load** `qa-context/modules/{MODULE}/module_memory.md`.
3. **Apply precedence:** module memory overrides project memory on conflict (Section 4.2). For machine rules, policy is the enforceable authority (Section 7).
4. **Treat `confirmed` assumptions as inputs; treat `proposed` ones cautiously and surface them.**

### 6.1 Mandatory citation in reports and handoffs

If `project_memory.md` exists, every report and handoff MUST include an explicit line stating that project memory was read — and, if a module is in scope and `module_memory.md` exists, that module memory was read too. Example lines:

```
Memory read: qa-context/project_memory.md (project) ✓
Memory read: qa-context/modules/ORD/module_memory.md (module ORD) ✓
Assumptions ledger: 6 entries (4 confirmed, 2 proposed) — 2 require human confirmation.
```

This citation is non-optional: a report that does not state memory was read is non-conforming.

### 6.2 Behavior when memory files are missing

- If `project_memory.md` / `project_assumptions.yaml` are **absent**, an agent must **not** silently proceed as if there are no rules. It must:
  - **On `policy-init`** (project setup) — create `project_memory.md` seeded with the verbatim default pilot content (Section 3.2) and create `project_assumptions.yaml` seeded with the example entries (Section 5.1).
  - **On `module-init`** (per-module setup) — create `qa-context/modules/{MODULE}/module_memory.md` from the module template (Section 4.1).
- If a run starts and the files still do not exist (init was never run), the agent must **note the absence explicitly in its report** ("WARNING: no project memory found; running with no persisted rules") and treat all interpretations as `proposed` / `needs_human_confirmation: true`.

(The exact command names `policy-init` and `module-init` and their full behavior are defined in `docs/v0.2_SPEC.md`; this section specifies only the memory-seeding obligation.)

### 6.3 Conflict reporting, not silent resolution

When the **frontend** (the implementation under test) conflicts with memory or policy, the agent **reports** the conflict — it does **not** silently resolve it (frontend override is forbidden). Any uncertain interpretation results in `needs_human_confirmation: true`, never an autonomous decision.

## 7. Interaction with Policy

Memory and policy are two layers with different jobs:

- **Memory** (`project_memory.md`, `module_memory.md`, `project_assumptions.yaml`) is **human narrative + assumptions**. It explains *why* and records *what we currently believe*, including uncertainty.
- **Policy** (`project_policy.yaml`, specified in `docs/v0.2_PROJECT_POLICY_SPEC.md`) is **enforceable machine rules**. It states *what is enforced* with no ambiguity.

**Authority on disagreement:** when memory and policy disagree, **policy is the enforceable authority** — agents follow the policy. However, the disagreement must **not** be hidden: the agent must **surface the memory-vs-policy conflict in its report for human resolution**, so the PM can either fix the memory or amend the policy. Policy winning at runtime never means the conflict goes unrecorded.

(Module-memory-vs-project-memory conflicts are resolved by module memory, Section 4.2; that is a memory-internal precedence and is separate from memory-vs-policy authority.)

## 8. Lifecycle Across Runs

Persistence only pays off if the PM updates memory after each run. The intended loop:

1. **Before a run** — agents read memory + assumptions (Section 6). Nothing is re-derived from scratch.
2. **During a run** — agents record new uncertain interpretations as `proposed` assumptions (`needs_human_confirmation: true`), and may add prose findings under "Open Questions" in the relevant memory file.
3. **After a dry run** (`docs/v0.2_DRY_RUN_GUIDE.md`) — the PM:
   - Reviews proposed assumptions and **confirms** or **rejects** each (Section 5.2).
   - Adds confirmed findings into the appropriate section of `project_memory.md` or `module_memory.md` as durable prose.
   - Records module-specific deviations as **overrides** in `module_memory.md` (Section 4.3).
4. **Next run** — the now-richer memory is read first, so the wrong assumption corrected last time never recurs.

This is the mechanism that **prevents repeated wrong assumptions**: a correction made once (confirm/reject/override) is written to a file and re-read on every future run, across sessions and project directories.

---

## 9. Provenance

Memory must not launder uncertainty into false certainty. Any inference that is **derived from memory** and then used in a screen contract, a defect report, or a handoff MUST still carry its provenance:

- `source` — point back to the originating memory/assumption (e.g. `project_assumptions.yaml#A-006` or `module_memory.md:ORD`).
- `confidence` — the confidence inherited from the assumption (or lower if the inference adds uncertainty).
- `needs_human_confirmation` — `true` if the underlying assumption is `proposed`, or if the inference involves any uncertain interpretation.

In particular: an inference built on a `proposed` assumption is itself uncertain and must be flagged `needs_human_confirmation: true`. Confirmed assumptions may be relied upon, but their `source` and `confidence` are still recorded so a reader can trace any contract claim back to the memory that justified it.
